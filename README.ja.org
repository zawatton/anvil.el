* Anvil とは？

Anvil は Emacs の機能を [[https://modelcontextprotocol.io/][Model Context Protocol (MCP)]] ツールとして公開し、
あらゆる AI コーディングアシスタントから呼び出せるようにします。
Claude Code、Codex CLI、Claude Desktop、GPT、Ollama/llama.cpp 経由のローカルモデルなど、
MCP を話すクライアントならどれでも対応しています。

~(anvil-enable)~ の一行で、Emacs が約 40 のツールを備えた (optional module を選んで有効化することで、用途に応じて拡張できます) AI 向け作業台に変わります。
ファイル編集、org-mode 操作、git 参照、Elisp 評価など、すべて安全なサンドボックス
インターフェース越しに利用できます。

* Emacs ユーザーじゃない方へ — トークン削減目的で導入する場合

「自分は Emacs を使わないけど、Claude Code / Codex CLI / Cursor のトークン消費を減らしたい」──
そういう動機でこのページに来た方向けの節です。

** Emacs の UI を開く必要はありません

Anvil は MCP server として裏で動くだけです。あなたが普段使っているエディタ
(VSCode / Cursor / Neovim / Claude Code / Codex CLI 単体 / その他) を変える必要はありません。

#+begin_example
  AI クライアント ──MCP──▶ Anvil (裏の Emacs daemon) ──▶ ファイル操作
  (VSCode / Cursor /                                       (差分のみ送受信)
   Claude Code / Codex CLI 等)
#+end_example

Emacs はバックグラウンドで daemon 1 つを常駐させるだけ。Emacs のウィンドウを
開いたり、=init.el= を書いたりする必要はありません。

** どこで効くか — 典型的な節約パターン

| シナリオ                                  | 削減率    | 何が起きているか                |
|-------------------------------------------+-----------+----------------------------------|
| 大きいファイルへの 1 行追加・削除          | *~90%+*   | フルファイル Read+Write を回避   |
| 同じファイルへの 3 件以上の編集            | *~70%*    | 1 ラウンドトリップ + atomic write |
| JSON / 設定辞書への N 件 key/value 追加   | *~73%*    | i18n、機能フラグ等               |
| TS/JS の ={ ja: "..." }= 形式データに英訳などを一括追加 | *~78%*    | マッピング JSON 1 回送信         |
| 巨大ファイルから繰り返し構造データ抽出     | *~50%*    | レガシー移行、ETL                |
| N ファイルの docstring を一括変更          | *~60-80%* | 1 コールで全ファイル分           |

数値は実測ベース ([[https://github.com/zawatton/anvil.el/blob/master/README.ja.org#効率性--anvil-が-ai-トークンを節約する理由][効率性節]] にケーススタディ)。
内容次第ですが、典型的なリファクタリングセッションで *15,000〜25,000 トークン* の
削減が安定して出ています。

** 最小セットアップ (Emacs 未経験前提)

*最短手順* — OS ごとに 1 コマンド。詳細は
[[file:docs/QUICKSTART.org][docs/QUICKSTART.org]] (Emacs 未経験者向け完全ガイド) 参照:

#+begin_src bash
# Linux / macOS
curl -fsSL https://raw.githubusercontent.com/zawatton/anvil.el/master/install.sh | bash
#+end_src

#+begin_src powershell
# Windows (PowerShell)
iwr -useb https://raw.githubusercontent.com/zawatton/anvil.el/master/install.ps1 | iex
#+end_src

インストーラーは Emacs の自動インストール (=apt= / =dnf= / =brew= /
=winget=)、anvil.el の clone、daemon 起動、MCP bridge 配置、
=~/.claude.json= への登録まで一気に行います。Emacs が既に入っている
なら =--skip-emacs= (Windows では =-SkipEmacs=) を付けて下さい。

*手動でやる場合*:

1. *Emacs を入れる* — daemon が動けば良いので普段使い設定は不要
   - Linux: =sudo apt install emacs= / =sudo dnf install emacs=
   - macOS: =brew install --cask emacs=
   - Windows: [[https://www.msys2.org/][MSYS2]] 経由 or [[https://www.gnu.org/software/emacs/download.html][gnu.org binary]]
2. *anvil を入れる* — [[https://github.com/zawatton/anvil.el/blob/master/README.ja.org#インストール][インストール]] 節の手順 (=async-installer= がもっとも手早い)
3. *daemon を起動して MCP server を上げる* — [[https://github.com/zawatton/anvil.el/blob/master/README.ja.org#クイックスタート][クイックスタート]] 節
4. *AI クライアントから接続* — [[https://github.com/zawatton/anvil.el/blob/master/README.ja.org#クライアント設定--ai-を-anvil-に接続する][クライアント設定]] 節 (Claude Code / Claude Desktop のスニペットあり)
5. *CLAUDE.md (or 相当) に Anvil 優先指示を入れる* — [[https://github.com/zawatton/anvil.el/blob/master/README.ja.org#推奨スニペットclaude-code-claudemd][推奨スニペット]] (任意だが効果大)

普段使う VSCode / Cursor / Neovim / Claude Code / Codex CLI はそのままで OK。これといって Emacs Lisp の知識を使うことなく、AI のトークン消費削減だけを得られます。

** 何がコストか — 正直な話

- *初回セットアップに 15〜30 分* — Emacs インストール + anvil clone + MCP 設定
- *Emacs daemon が常駐* — 平時 50〜200 MB 程度のメモリ
- *Emacs を新しい依存として持つ* — アップデートや OS 移行時に 1 つ増える

これらと引き換えに得られるのが上記表の節約です。エージェント作業時間が
1 セッション 30 分以上、月数回以上ある人なら、初回コストはほぼ即座に
回収できます。

* なぜ Anvil か？

- *LLM 非依存* — MCP 対応クライアントならどれとでも動作
- *安全設計* — 時間のかかる作業を裏の別 Emacs に逃がし、対話中の Emacs に影響を与えない
- *org-mode ネイティブ* — org の読み書き・リファクタを一級サポート
- *大容量ファイル対応* — 1.2 MB 超のファイルもデータ損失・切り捨てなしで処理
- *クロスプラットフォーム* — Linux、macOS、Windows、WSL
- *モジュラー* — 必要なものだけロード、独自モジュールで拡張可能

* 効率性 — Anvil が AI トークンを節約する理由

LLM エージェントはトークン課金です。Read/Edit のたびにバイトが飛び、
オーケストレーションのオーバーヘッドが積み重なります。Anvil のツールは
エージェントのコンテキスト予算を、書類整理ではなく推論に使えるよう設計されています。

** Manifest プロファイル — セッション manifest コスト (v0.4.0)

Doc 26 の =anvil-manifest= は、クライアントが virtual server-id
(=emacs-eval-ultra= / =-nav= / =-core= / =-agent= / =-edit=) 経由で
接続した際に =tools/list= をフィルタします。hidden tool も =tools/call=
で明示呼出可能なのでハンドラ自体は live、セッション冒頭で Claude Code
が読む manifest 前置きだけが縮みます:

| プロファイル | 公開ツール数 | 概算トークン | 用途                               |
|---------+--------------+--------------+------------------------------------|
| =full=  | 198          | ~20,400       | 全機能                             |
| =core=  | 54           | ~5,800        | 日常コーディング、memory エンジン off |
| =lean=  | 59           | ~6,620        | memory エンジン off                |
| =nav=   | 28           | ~3,100        | read-only 探索                     |
| =ultra= | 17           | ~2,740        | hot tool + =ts_extended= lazy       |
| =agent= (v0.4.0) | ≈60 (intent filter) | ~6,500 | orchestrator / session / memory / edit |
| =edit=  (v0.4.0) | ≈40 (intent filter) | ~4,200 | ファイル / org / code / json 編集専用  |
| =dynamic= (2026-05) | Top-K + warm LRU | query 依存 | Tool Attention + schema trimming |

=manifest-cost= MCP ツールを叩けば現状の profile / advertised-count /
registered-count / approx-tokens がいつでも取得できます。

2026-05 update では、この仕組みが Tool Attention layer に拡張されています。
=ANVIL_PROFILE=dynamic= と、必要に応じて =ANVIL_ATTENTION_QUERY= を設定すると
現在の意図で tool が rank されます。=:preconditions= が =ANVIL_STATE_TAGS= に
合わない tool は隠され、schema trimming 有効時は Top-K の focused tool、
manifest 自身の tool、最近使った warm LRU tool だけが full schema を持ちます。
それ以外は name/type 中心の compact schema で広告されます。
=manifest-recovery-suggest= は tool が隠れた理由を説明し、
=manifest-docstring-candidates= は弱い説明文を Doc43 autoresearch へ渡します。

*Orchestrator 子セッション* (Doc 26 Phase 1b + Doc 34 Phase B) は
=anvil-orchestrator-manifest-profile= を設定するだけで、自動生成された
=--mcp-config= が =emacs-eval-PROFILE= 仮想 server に接続します。
実測: 20 provider の合意を 200 子セッションで回すと =full= で
manifest だけで約 4M tokens を消費するのに対し、=ultra= では約 550k ──
*manifest 単独で ~87% 削減* (エージェントが一言発する前の段階で)。

** ファイル全体ではなく、領域に操作する

汎用ファイルツールは Read → Edit のパターンを強制し、そのたびにファイル全体を
転送します。Anvil は差分だけを送る、ピンポイントな操作を提供します。

| タスク | 汎用フロー | Anvil フロー |
|--------+------------+--------------|
| 1.2 MB の org ファイルで文字列を 1 件置換 | 1.2 MB を Read → Edit (旧文字列 + 新文字列を送信) | =anvil-file-replace-string= (パターンと置換文字列のみ) |
| 3 万行の =init.org= から見出し 1 件を読む | Read してページネーション | =anvil-org-read-headline= (サブツリーのみ) |
| 大きいファイルの末尾付近に挿入 | ファイル全体を Read → 書き直し | =anvil-file-insert-at-line= (差分のみ) |

org-mode のリファクタ操作（セクション移動、refile、分割）に限れば、
MCP 経由の操作は Read + Write 方式に比べて *トークンを 10〜20 倍節約* できます。

** 複数の編集を 1 往復にまとめる

エージェントが同じファイルに複数回編集を加えると、それぞれが独立したツール呼び出しに
なり、リクエスト・スキーマ・レスポンスのオーバーヘッドが重複します。
=anvil-file-batch= は編集ベクタを 1 回のトランザクションで適用し、
個別に発行する場合に比べて *最大 ~90% のトークン削減* を実現します。

** 重い処理は帯域外で実行

低速な操作（大規模タングル、フルツリー org スキャン、バイトコンパイル）は
ワーカーデーモンプールにディスパッチされるため、エージェントのメイン接続は
常に応答可能な状態を保ちます。

実測値: ~1 MB の =init.org= に対する =org-babel-tangle= はインプロセスで ~30 秒
かかりますが、ワーカーディスパッチなら ~2 秒で完了します。*エンドツーエンドで
~15 倍高速*、かつエージェントのメインセッションが一切ブロックされません。

** 大規模 org データベースへのスケール

テンプバッファをウォークする素朴なアプローチは、MCP リクエスト内で ~50 KB を超える
org ファイルに対してタイムアウトします。Anvil の org ツールはストリーミング正規表現
スキャナを使うため、マルチメガバイトの org ツリー（パワーユーザーが実際に運用している
モノリシックな =init.org= やジャーナルファイル）でも見出し検索・アウトライン読み出しが
高速に保たれます。

** スタンドアロン runtime — Emacs daemon 起動コストが消える (v1.0)

NeLisp Doc 44 の =bin/anvil mcp serve --no-emacs= は、Emacs daemon
ではなく =anvil-runtime= Rust binary を起動します。short-lived な
MCP セッション (CI 実行、one-shot エージェント dispatch、ephemeral
container) では、Emacs daemon の cold-start を完全に飛ばし、フル
装備 Emacs daemon に比べて常駐プロセスがずっと小さく収まります。

Architecture α (=anvil-http= / =anvil-state= / =anvil-defs= /
=anvil-org-index= が =fboundp= ガード + フォールバックで NeLisp に
core helper を委譲) により、同じ MCP ツール surface が両基盤から
到達可能です:

| デプロイ形態      | ツール surface                                  |
|-------------------+--------------------------------------------------|
| Emacs 同梱        | =anvil.el= フル + =anvil-ide.el=                |
| Rust のみ         | =anvil.el= フル (IDE は subprocess fallback)    |

既存 Emacs ユーザーは =bin/anvil mcp serve= をそのまま継続。
Container / ephemeral runner / 「Emacs を学ばずに AI workbench
だけ欲しい」ユーザーは、=apt install emacs= 不要で同じ MCP ツール
surface を Rust 経路から得られます。

** ケーススタディ: 大規模 TypeScript リファクタリング

Anvil は理論だけの話ではなく、org-mode 専用でもありません。実際の大規模
TypeScript 移行プロジェクト — [[https://github.com/zawatton/newDTW.github.io][newDTW]] リポジトリ（デコンパイルされた HSP
アプリケーションを TypeScript に移植）— で検証されています。

- *1,400+* モジュールにわたる *16,000+* 行の TypeScript
- 実装ファイルと密結合した巨大な型定義ファイル（数千行規模）
- 同一ファイルへの多点編集が高頻度で発生

org-mode で効果を発揮した Anvil ツールが、ここでも定量的な成果をもたらしました。

*** =anvil-file-batch= — 1 ファイルへの複数同時編集

TypeScript のリファクタリングでは、1 ファイルに対してインポート追加・型定義変更・
実装置換・メソッドリネームを同時に行うことが日常的です。

実測: 130 行のソースファイルへの 6 編集。

| 方法                  | トークン | ラウンドトリップ |
|-----------------------+----------+------------------|
| 汎用 =Edit= ×6        | ~4,800   | 6                |
| =anvil-file-batch=    | ~1,500   | 1                |
| 削減率                | *~70%*   | *83%*            |

*** =anvil-file-replace-regexp= — 型・識別子の一括リネーム

=Promise<T>= → =AsyncResult<T>= へのコードベース全体の置換や、グローバル変数の
リネームを *1 コマンド* で完結できます。従来フロー（数百ファイルに grep し、個別に
read して edit）の機械的な繰り返しを一掃します。

*** =anvil-file-insert-at-line= / =-delete-lines= — 大きいファイルへの局所操作

ユースケース例: 4,000 行超の型定義ファイルへのフィールド 1 件追加、
またはグローバルステートインターフェースへの追記。

| 方法                         | トークン   |
|------------------------------+------------|
| ファイル全体を Read + Edit   | 数千       |
| =anvil-file-insert-at-line=  | 数十       |
| 削減率                       | *~90%+*    |

*** =anvil-json-object-add= — i18n / 設定辞書への一括追加

言語辞書、機能フラグ、設定マップなど、JSON オブジェクトに
多数の key/value を追記するワークフローでは、汎用の =Edit=
ツールを使うとコールごとに長いアンカー文字列を送出する必要
があります。

実測: i18n キー約 200 件を 40 件ずつ 5 バッチで追加。

| 方法                          | トークン | コール |
|-------------------------------+----------+--------|
| 汎用 =Edit= ×5                | ~30,000  | 5      |
| =anvil-json-object-add= ×5    | ~8,000   | 5      |
| 削減率                        | *~73%*   | 0      |

カンマ、閉じ波括弧、インデント検出、重複キー対応は自動。

*** =anvil-file-ensure-import= — import 行の冪等追加

多数のソースファイルに =import= 行を追加する作業は頻出
パターンですが、既に存在するかを毎回確認する手間があります。
=file-ensure-import= は対象行の存在確認、無ければ最後の
ヘッダ行の直後に 1 行挿入、の流れを 1 コールに集約します。
=:after-regex= を指定すれば Emacs の =(require ...)= 列にも
同じように使えます。

*** =anvil-file-batch-across= — 複数ファイルを 1 コールでリファクタ

N ファイルに似た変更 (docstring 更新、リネーム、import 追加)
を加えるリファクタでは、各ファイルの =operations= 配列を
まとめて送ることで全工程を 1 MCP コールで完了できます。

*** =anvil-code-extract-pattern= — 巨大ファイルからの構造データ抽出 (read-only)

レガシーコードからのデータ抽出や ETL でよくある「ファイル全体を Read
してから regex で組み立てる」を、1 MCP コールに置き換えます。
=:block-start= でブロック開始、=:block-end= でブロック終端 (=brace-balance=
は文字列内の波括弧を skip)、=:fields= 各 regex で値を捕捉、まとめて
プレーンデータで返します。ファイル本体は読み出されません。

実測: newDTW の =func492.ts= (3,407 行) から 347 件の if-block アイテム
定義を抽出。

| 方法                                | トークン     |
|-------------------------------------+--------------|
| ファイル全体 Read + 抽出スクリプト  | ~8,000 + α   |
| =anvil-code-extract-pattern= ×1     | ~3,000-5,000 |
| 削減率                              | *~50%*       |

レガシーマイグレーション、データ移行、i18n 棚卸しなど、繰り返し
構造を持つテキストを扱う作業全般で効きます。

*** =anvil-code-add-field-by-map= — TS/JS データに多言語訳文を一括追加

i18n 翻訳追加のような「同じファイルの全 ={lookup-key: "..."}= に
={add-key: "..."}= を一括で足したい」作業を、マッピング JSON を
1 回送るだけに短縮します。デフォルトは dry-run (=:apply= 既定 nil)
なので、まず preview を見てから =:apply t= で書き込む 2 段階で
安全に運用できます。

実測: newDTW の data ファイル 8 本 (各 25-100 entry) に =en= 翻訳を追加。

| 方法                                          | トークン |
|-----------------------------------------------+----------|
| 各ファイル Read + 全文置換 Write              | ~16,000  |
| =anvil-code-add-field-by-map= (マッピングのみ) | ~3,200   |
| 削減率                                        | *~78%*   |

既存 add-key の扱いは =:on-existing= で制御 (既定 =error= で fail-loud)、
誤上書きを防ぎます。

*** TypeScript が特に恩恵を受ける理由

| TypeScript の特性                         | Anvil が助ける理由                              |
|-------------------------------------------+--------------------------------------------------|
| 型定義と実装が同一ファイルに共存           | 1 ファイルへの多編集 → =file-batch= が輝く      |
| 巨大な型定義ファイル                       | 局所操作でフルファイル read を回避              |
| 頻繁なインポート追加                       | =file-insert-at-line= が定型追加に対応          |
| グローバルステートインターフェース         | フィールド追加はパターン化しやすく、スクリプト化も容易 |
| レガシー if-block / switch からの抽出     | =code-extract-pattern= が Read 全文を不要にする |
| TS/JS データに多言語訳文を一括追加          | =code-add-field-by-map= がマッピング 1 回で完結 |

newDTW の 1 リファクタリングセッションでの累積節約は *15,000〜25,000 トークン* に
のぼりました (~code-extract-pattern~ / ~code-add-field-by-map~ 投入後)。節約分は
ファイル転送ではなく、実際の推論に使われました。

** Anvil を使うべき場面

| シナリオ                                       | Anvil の効果                         |
|------------------------------------------------+--------------------------------------|
| 同一ファイルへの 3 回以上の編集               | ◎ 大幅なトークン削減                |
| 大きいファイルへの局所的な追記                | ◎ フルファイル Read を回避          |
| 型・識別子の一括リネーム                      | ◎ 1 コマンドで完結                  |
| インポート追加 / 型フィールド追記             | ○ ラウンドトリップ削減              |
| エージェント作業中の重い Emacs 操作           | ◎ ワーカーディスパッチ (~15× 高速)  |
| 新規ファイルのスクラッチ作成                  | △ 汎用の =Write= で十分             |
| ファイル・リポジトリの探索                    | × Glob / Grep の方が適切            |

** 実際のところどうなるか

大きな設定ファイルを触り、いくつかの org 見出しをリファクタし、タングルを走らせる
典型的なエージェントセッションは、本来ファイル転送オーバーヘッドに支配されます。
Anvil を使うと:

- 各編集は *変更バイトだけ* を送る。ファイル全体ではない。
- 多段階リファクタは *1 回* のバッチ呼び出しに折り畳まれる。
- 低速な Emacs 操作はエージェントのスレッドではなく *ワーカー* で実行される。

結果として、トークンあたりの有効仕事量が増えます — コンテキスト予算がコードの
再読みではなく、コードについての推論に使われます。

** バッファ優先編集 — anvil-modes-allow-buffer-modify

デフォルトでは、anvil の org 編集ツール (update-todo-state, add-todo,
rename-headline, edit-body) は *ディスク優先* ポリシーを採用しています。
ファイルをテンポラリバッファに読み込み、編集を適用し、結果をディスクに書き出し、
訪問中のバッファがあればリフレッシュします。いずれかのバッファに未保存の変更がある
場合は操作が拒否されます — anvil は人間が作業中の編集を上書きしないからです。

=anvil-modes-allow-buffer-modify= に major mode のリストを設定すると、
オープン中のバッファがその mode にマッチするファイルについて、anvil は
*バッファ優先* ポリシーに切り替えます:

1. ファイルを訪問している最初のバッファを見つけます
   (indirect バッファは base バッファを辿ります)。
2. バッファに既に未保存の変更があるか確認します。
3. ライブバッファ内で直接編集を行います。
4. 編集前にバッファがクリーンだった場合 *のみ* 保存します —
   作業中の編集を保持するため、未保存の変更があった場合は保存しません。

つまり、AI エージェントと人間が同じファイルで共同作業する際、
エージェントが書きかけの編集を強制保存することはありません。

#+begin_src emacs-lisp
;; org ファイルのみバッファ優先編集を有効化
(setq anvil-modes-allow-buffer-modify '(org-mode))

;; または、全オープンバッファで有効化
;; (fundamental-mode は特別扱いで、あらゆる major mode にマッチします)
(setq anvil-modes-allow-buffer-modify '(fundamental-mode))
#+end_src

** Browser モジュール（オプション）

=browser= モジュールは [[https://agent-browser.dev/][agent-browser]] CLI をラップし、ページのアクセ
シビリティツリー要約を MCP tool 1 call で取得できるようにします
（生 DOM dump 比で typically 25× トークン節約）。

** Memory モジュール（オプション）— AI CLI 横断の auto-memory インデックス

=memory= モジュールは disk 上の =.md= auto-memory ファイルを
provider-agnostic に一括索引する opt-in モジュールです。Claude
Code / Codex CLI / Gemini CLI / Continue / Aider / Cline /
Cursor など、どの MCP クライアントからも同じ長期コンテキストを
共有できます。

有効化:

#+begin_src emacs-lisp
(add-to-list 'anvil-optional-modules 'memory)
(anvil-enable)   ; モジュール群を再読み込み
#+end_src

disk 上の既存ファイルを一括インデックスする:

#+begin_src text
;; Emacs から:
M-x anvil-memory-scan

;; MCP クライアントから:
`memory-scan' ツールを引数なしで呼ぶ
#+end_src

=memory-scan= は =anvil-memory-provider-paths= (下記既定) がマッチ
する全ディレクトリを walk し、=.md= 1 ファイルにつき 1 行投入して
FTS5 body 索引を更新します (=MEMORY.md= はスキップ)。冪等なので
何度でも再実行可能 — 既存行の =access_count= / =validity_prior= は
ON CONFLICT で保持されます。

プロバイダ既定値 (追加・編集可能):

#+begin_src emacs-lisp
(setq anvil-memory-provider-paths
      '((claude   . "~/.claude/projects/*/memory")
        (codex    . "~/.codex/memories")
        (gemini   . "~/.gemini/memories")
        (continue . "~/.continue/memories")
        (aider    . "~/.aider/memories")
        (cline    . "~/.cline/memories")
        (cursor   . "~/.cursor/memories")
        (windsurf . "~/.windsurf/memories")
        (zed      . "~/.config/zed/memories")))
#+end_src

パターンは =*= glob を含めて良く、存在しないパスは silently
skip されます (未インストールの provider を追加してもコスト 0)。
=anvil-memory-roots= を明示的に list にすると auto-detect を完全
override できます (テスト / サンドボックス用途)。

インデックス後は 1 つの SQLite DB
(=~/.emacs.d/anvil-memory-index.db=) 上で全プロバイダ横断に通常
tool が使えます:

- =memory-search QUERY= — FTS5 全文検索 (=:limit= / =:type= 指定可)
- =memory-audit= — TTL チェック。=:with_urls t= で
  reference URL を =anvil-http-head= 生存確認
- =memory-list= — =:with-decay t :sort 'decay= で decay スコア
  降順
- =memory-save-check SUBJECT BODY= — 書き込み前の Jaccard 類似
  top-N 候補照合 (プロバイダ横断)
- =memory-duplicates= — 横断ニア重複ペア検出
- =memory-promote OLD NEW-TYPE= — rename + 索引更新
- =memory-regenerate-index= — decay 降順で =MEMORY.md= body 生成
  (read-only 出力、実際の write は呼び出し側の責任)
- =memory-add= / =memory-get= / =memory-export-md= — DB-primary な
  write/read/export 経路。新規 memory を SQLite に直接追加し、
  backing =.md= を直接 read せずに本文取得し、他 AI と file で
  受け渡す必要がある時だけ Markdown に戻せます。
- =memory-session-delta= / =memory-stop-sync= — WHISPER session context。
  初回は decay 順の full index、以後は追加・変更・削除分だけを返し、
  delta が大きくなりすぎた場合は full に戻ります。
- =memory-harvest-candidates= — session text / transcript から永続化候補を
  review-only で抽出。自動保存はせず、どれを =memory-add= するかは
  呼び出し側が決めます。

Claude 以外の MCP クライアントは同じ anvil サーバに接続すれば
これらの tool を呼べます。通常の scan/search/list/audit は provider
管理の =.md= に対して read-only です。変更は明示的な操作
(=memory-promote= の rename、=memory-add= の DB insert、
=memory-export-md= の caller-driven Markdown export) に限られます。

*** DB 配置とマシン間共有

metadata DB の既定値は =<user-emacs-directory>/anvil-memory-index.db= です。
notes repo 配下の DB を使いたい場合は symlink ではなく root 指定で
redirect できます:

#+begin_src emacs-lisp
(setq anvil-memory-shared-db-roots
      '("~/Cowork/Notes"))
#+end_src

=anvil-memory-db-path= が既定値のままなら、DB path は初回 access 時に
次の順で解決されます:

1. =ANVIL_MEMORY_DB= environment variable
2. =anvil-memory-db-path= (明示 customize 時)
3. =.anvil-memory/anvil-memory-index.db= を含む
   =anvil-memory-shared-db-roots= の最初の entry
4. 既定の =<user-emacs-directory>/anvil-memory-index.db=

実際の path は =(anvil-memory-effective-db-path)= で確認できます。

複数マシンで 1 つの SQLite file を git merge しないでください。
現在の運用モデルは:

- =.anvil-memory/*.db= は machine ごとに保持し、gitignore する
- DB-direct memory を他 AI に渡す必要がある時だけ =memory-export-md=
  で Markdown artifact にする
- 別マシンでは =memory-scan= で export 済み =.md= を再 index する

これにより binary merge conflict と、長時間 Emacs daemon が古い
SQLite handle を持ち続ける問題を避けます。

*** Memory bridge — cross-AI / mobile access

=memory-bridge= optional module は =anvil-memory= と =anvil-worklog= の
前段に localhost JSON API を置きます。Emacs MCP server を直接話せない
browser chat UI、mobile thin client、Tailscale 越しの別マシン、
read replica から使うための層です。

#+begin_src emacs-lisp
(add-to-list 'anvil-optional-modules 'memory)
(add-to-list 'anvil-optional-modules 'worklog)
(add-to-list 'anvil-optional-modules 'memory-bridge)
(anvil-enable)
(anvil-memory-bridge-start)
#+end_src

既定 bind は =127.0.0.1:8730= です。read endpoint は
=/memory/search=、=/memory/list=、=/memory/get/<name>=、
=/memory/events=、=/worklog/search=、=/worklog/list=、=/worklog/get=。
memory の mutate endpoint は =Authorization: Bearer <token>= が必要で、
=M-x anvil-memory-bridge-rotate-token= で token を rotate できます。

=extensions/= 配下には thin client があります:

- =extensions/memory-bridge/= — Claude.ai / ChatGPT / Gemini 用の
  Manifest V3 browser extension。=Ctrl+Alt+M= で memory 検索・注入、
  =Ctrl+Alt+S= で選択 text を保存。
- =extensions/mobile-bridge/= — 依存なしの static mobile app。
  bridge URL/token を =localStorage= に保存し、失敗した save は
  online 復帰時に replay します。

read-replica mode は single-writer 前提です。replica は primary の
=memory_event= row を pull できますが、local mutate endpoint は =409=
を返し、複数マシンが silently diverge しないようにします。

** Worklog モジュール（オプション）

=worklog= モジュールは long-term memory とは別に AI 作業ログを index します。
legacy な =capture/ai-logs-*.org= archive は scan できますが、新規 entry は
org 手動追記ではなく =worklog-add= による DB-primary です。search/list/get は
MCP tool と memory-bridge の両方から使えます。

=memory= と同じく、=worklog= も machine ごとの SQLite DB を前提にします。
git や他マシンとの受け渡しには =worklog-export-org= で human-readable な
org snapshot を作ります。2026-05 の develop では、長時間 daemon が
SQLite file の差し替え・unlink を検出し、次の DB 操作前に自動 reopen します。

典型的な使い方:

#+begin_src text
worklog-add(title, body)
worklog-search("Doc43 autoresearch")
worklog-get("anvil-worklog:db:linux-debian:2026", 1779295660)
worklog-export-org(machine, year)
#+end_src

* クライアント設定 — AI を Anvil に接続する

Anvil が Emacs 内で起動したら、AI クライアント（Claude Desktop、Claude Code CLI、Codex CLI、
その他 MCP 対応クライアント）に MCP サーバとして登録する必要があります。

ブリッジは小さなシェルスクリプト (=anvil-stdio.sh=) で、stdio 上で JSON-RPC を話し、
=emacsclient= 経由で動作中の Emacs デーモンに呼び出しを転送します。
マシンごとに一度インストールします:

#+begin_src
M-x anvil-server-install
#+end_src

デフォルトでは =anvil-stdio.sh= を =user-emacs-directory=（通常 =~/.emacs.d/=）に配置します。
配置先は =M-x customize-variable RET anvil-server-install-directory= で変更できます。

*前提条件:* Emacs がデーモンとして起動していること
(=server-start= または =emacs --daemon=)。Anvil は =emacsclient= 経由で接続します。

** Claude Desktop

設定ファイルの場所:

| OS                | パス                                                                                        |
|-------------------+---------------------------------------------------------------------------------------------|
| macOS             | =~/Library/Application Support/Claude/claude_desktop_config.json=                           |
| Linux             | =~/.config/Claude/claude_desktop_config.json=                                               |
| Windows (通常)    | =%APPDATA%\Claude\claude_desktop_config.json=                                               |
| Windows (UWP)     | =%LOCALAPPDATA%\Packages\Claude_<id>\LocalCache\Roaming\Claude\claude_desktop_config.json= |

Anvil は現在、内部的にツールを *2つの MCP server-id* に分けて
登録しています — =anvil=（eval / IDE 系）と =emacs-eval=（file,
org, buffer, worker など）。多くのクライアントは =mcpServers=
エントリ1つあたり1つのツールリストしか表示しないため、すべての
ツールを公開するには、同じスクリプトを別々の =--server-id= 値で
指す *2エントリ* が必要です。（将来のリリースでは server-id を
1つに統合する予定。当面は2エントリ構成がサポート設定です。）

*** macOS / Linux

#+begin_src json
{
  "mcpServers": {
    "anvil": {
      "command": "/path/to/.emacs.d/anvil-stdio.sh",
      "args": [
        "--server-id=anvil",
        "--init-function=anvil-enable",
        "--stop-function=anvil-disable"
      ]
    },
    "anvil-emacs-eval": {
      "command": "/path/to/.emacs.d/anvil-stdio.sh",
      "args": [
        "--server-id=emacs-eval"
      ]
    }
  }
}
#+end_src

2つ目のエントリは既に起動中のデーモンに接続し、file/org/buffer/
worker 系ツールを公開します。=--init-function= は1つのエントリに
だけ指定すればOK — 最初の接続で anvil が enable され、両方の
server-id にツールが登録されます。

Emacs デーモンがデフォルト以外のソケットディレクトリを使用している場合は、
*両方* のエントリの =args= に =--socket=/path/to/server= を追加してください。

*** Windows

Windows では Git Bash がシェルスクリプトの実行に必要です:

#+begin_src json
{
  "mcpServers": {
    "anvil": {
      "command": "C:\\Program Files\\Git\\bin\\bash.exe",
      "args": [
        "C:/Users/<you>/.emacs.d/anvil-stdio.sh",
        "--server-id=anvil",
        "--init-function=anvil-enable",
        "--stop-function=anvil-disable"
      ]
    },
    "anvil-emacs-eval": {
      "command": "C:\\Program Files\\Git\\bin\\bash.exe",
      "args": [
        "C:/Users/<you>/.emacs.d/anvil-stdio.sh",
        "--server-id=emacs-eval"
      ]
    }
  }
}
#+end_src

設定編集後は Claude Desktop を再起動してください。

** Claude Code (CLI)

CLI の設定は =~/.claude.json= にチャット履歴などと混在して保存されています。
*ファイル全体を上書きしないでください*。=claude mcp add= コマンドで
サーバエントリを追加します。すべてのツールを公開するには *両方* の
コマンドを実行してください:

#+begin_src sh
claude mcp add -s user -t stdio anvil -- \
  ~/.emacs.d/anvil-stdio.sh \
  --server-id=anvil \
  --init-function=anvil-enable \
  --stop-function=anvil-disable

claude mcp add -s user -t stdio anvil-emacs-eval -- \
  ~/.emacs.d/anvil-stdio.sh \
  --server-id=emacs-eval
#+end_src

Windows の場合は、スクリプトパスを上記の Git Bash 呼び出し形式に置き換えてください
(=command= = bash.exe、最初の =arg= = スクリプトパス)。

** Codex CLI

Codex CLI も同じ MCP stdio bridge を使えます。Claude Code と同じく、
初期化用の =anvil= entry と、file/org/buffer/worker および
=shell-run= / =shell-filter= / =context-compress= / =context-retrieve= /
=context-stats= などを公開する =anvil-emacs-eval= entry の *両方* を
登録してください。

=codex mcp add= で MCP server を管理する場合:

#+begin_src sh
codex mcp add anvil -- \
  ~/.emacs.d/anvil-stdio.sh \
  --server-id=anvil \
  --init-function=anvil-enable \
  --stop-function=anvil-disable

codex mcp add anvil-emacs-eval -- \
  ~/.emacs.d/anvil-stdio.sh \
  --server-id=emacs-eval
#+end_src

Codex のバージョンが project/global =config.toml= 管理の場合は、
同じ command / args の stdio MCP entry を作ってください。重要なのは
=emacs-eval= entry が存在することです。圧縮系ツールはそこに登録されます。

** オプション: ワーカープールも登録する

帯域外実行が有効なワークロード（大規模タングル、マルチ MB org スキャン、
バイトコンパイル）向けに、Anvil のワーカーモジュールはサブ Emacs デーモンを起動します。
AI クライアントから直接アドレスできるよう、別の MCP サーバとして登録することもできます:

#+begin_src json
{
  "mcpServers": {
    "anvil": { ... メインデーモン ... },
    "anvil-worker": {
      "command": "/path/to/.emacs.d/anvil-stdio.sh",
      "args": [
        "--socket=/path/to/.emacs.d/server/anvil-worker",
        "--server-id=anvil-worker"
      ]
    }
  }
}
#+end_src

ほとんどのユーザーはこれは不要です。Anvil の =worker= モジュールがメイン接続越しに
重い操作を自動ディスパッチします。

** 接続の確認

Emacs 内で:

| コマンド                      | 表示内容                                          |
|-------------------------------+---------------------------------------------------|
| =M-x anvil-describe-setup=    | ツール、リソース、クライアント設定例              |
| =(anvil-server-status)=       | メイン MCP サーバの状態（起動中 / 停止中）       |
| =(anvil-worker-status)=       | ワーカープールの状態                              |

AI クライアント側: エージェントにツール一覧を要求するか、読み取り専用の
=anvil-host-info= などを呼び出してみてください。呼び出しが返れば、ブリッジは
正常に動作しています。

* AI に Anvil を使わせる — AGENTS.md / CLAUDE.md の書き方
  
Anvil を接続しただけではツールが「使える状態になる」だけで、
エージェントが組み込みの =Read= / =Edit= / =Write= より Anvil を
*優先する* ようにしないと、上で説明したトークン節約は実現しません。
ほとんどの LLM クライアントはプロジェクト単位の指示ファイル
（Codex なら =AGENTS.md=、Claude Code なら =CLAUDE.md=、Cursor なら =.cursorrules= 等）
を読むので、以下のスニペットを貼り付けてください。

** 推奨スニペット（=AGENTS.md= / Claude Code =CLAUDE.md=）

#+begin_src markdown
## ファイル編集

該当する場合は、組み込みの Read/Edit/Write より Anvil MCP ツールを
優先する。差分のみを送り、複数編集を 1 ラウンドトリップに集約し、
フルファイル read を回避できる。

- `anvil-file-batch` — 同一ファイルへの 3 つ以上の編集（1 コールに集約）
- `anvil-file-replace-string` / `anvil-file-replace-regexp` —
  ピンポイント置換。ファイル全体を read 不要
- `anvil-file-insert-at-line` / `anvil-file-delete-lines` /
  `anvil-file-append` — 行単位の局所操作

組み込みの `Edit` は小さな単発編集のみに使う。同一ファイルに 3 箇所
以上編集するなら必ず `anvil-file-batch` を使う。

## org-mode

セクション移動・refile・分割、または大きい org からの単一見出し読み取りは、
Read+Write ではなく `anvil-org-*` ツールを使う。トークン消費が 10〜20 倍
少ない。

- `anvil-org-read-headline` — 単一サブツリーの読み取り
- `anvil-org-read-outline` — body なしのアウトライン表示
- `anvil-org-edit-body` / `anvil-org-rename-headline` /
  `anvil-org-update-todo-state` — 局所的な org 編集

## 重い処理 — ワーカーディスパッチ

長時間の Emacs 操作（大規模タングル、バイトコンパイル、マルチ MB
org スキャン、フルツリー検索）はメインデーモンで実行してはいけない。
他のすべてのツール呼び出しがブロックされる。ワーカープール経由で
ディスパッチすること。

- Anvil 内から呼ぶ elisp: ~1 秒を超えそうなものは raw `eval` ではなく
  `anvil-worker-call` を優先する。
- ワーカーを別 MCP サーバとして登録している場合（README 「オプション:
  ワーカープールも登録する」参照）、重い `eval` は
  `mcp__anvil-worker__eval` を直接ターゲットにする。メインセッションの
  応答性が保たれる。

ワーカー使用すべきだったサイン: メイン MCP セッションが数秒間ツール
呼び出しを受け付けなくなった。

## スケジュールタスク（cron）

`anvil-cron` タスクが設定されている場合（lint、ヘルスチェック、バッチ
インデクサ等）、その作業を場当たり的に再実装しない。cron MCP ツール経由で
状態確認・トリガーする:

- `anvil-cron-list` — 登録タスクとスケジュール一覧
- `anvil-cron-status` — 最終実行時刻、状態、最近の失敗
- `anvil-cron-run` — 登録済みタスクの即時実行

新規アドホックスクリプトを書く前に `anvil-cron-list` を確認する。
すでにジョブとして定義されているかもしれない。

## context / output 圧縮

コマンド出力や取得済み context が長い場合は、main reasoning loop に
戻す前に圧縮する。

- shell コマンドには `shell-run` を使う。stdout が自動filterされ、
  圧縮済みstdoutと `tee-id` が返る。圧縮表示で足りない場合だけ
  `shell-tee-get` で raw stdout を復元する。
- shell以外の長文には `context-compress` を使う。API JSON、RAG snippet、
  Web/article抽出、別tool由来のlog、diff、code断片が対象。後で原文が
  必要になりそうなら `store=true` にし、返ってきた `ccr-id` を
  `context-retrieve` に渡して復元する。
- 圧縮層が効いているかは、勘ではなく `context-stats` / `shell-gain` で
  確認する。

法令、金融、安全、正確な数値、契約文言など、厳密な文面に依存する作業では
圧縮表示だけを根拠にしない。必要な主張をする前に raw context を復元する。
#+end_src

** 自己強化ルール（任意）

長いセッションでも軌道修正できるよう、自己補正トリガを追記しておく:

#+begin_src markdown
## MCP ツール自己強化

作業中に以下を検知したら、続行前に該当する Anvil ツールへ切り替える:

- 同じ elisp パターンを同一セッション内で 2 回以上書いた
- 1 つの論理編集に対して `anvil-eval` を 3 回以上呼んだ
  （`anvil-file-batch` 1 コールで済んだはず）
- 同じ大きいファイルをフルで何度も Read している
- 重い elisp 操作でメインセッションがブロックされた
  （`anvil-worker-call` / `mcp__anvil-worker__eval` 経由にすべきだった）

タスク末尾ではなく、その場で軌道修正すること。
#+end_src

** なぜこれが必要か

明示的な指示がないと、エージェントはどの MCP サーバが接続されていても
最大公約数的なツール（=Read=、=Edit=、=Write=）にフォールバックします。
README 冒頭のトークン節約表は、エージェントが実際に Anvil ツールを
*選ぶ* ことを前提にしています。その判断を駆動するのは指示ファイルで
あって、Anvil 自身ではありません。

* FAQ — 他のパッケージとの違いは？

** …claude-code-ide との違い?

両者は異なる問題を解決していて、併用できます。
=claude-code-ide= は
Claude Code のインタラクティブ UI を eat 経由で Emacs の /中に/
持ち込みます。MCP サーフェスは主に IDE 寄り (xref / diagnostics /
buffer metadata)。ファイル編集は Claude Code 組み込みの Read / Edit /
Write をファイルシステム直で走らせます。

anvil はその逆です: スタンドアロンの MCP サーバとして /Emacs 自体/
— file 操作、org primitives、worker dispatch、elisp introspection —
を任意の MCP クライアントに公開します。実務では私は両方動かしていて、
claude-code-ide で Emacs 内インライン Claude セッション、anvil は
別 MCP サーバとして claude-code-ide /と/ スタンドアロンの ~claude~ CLI
両方から呼ばれます。

** …「任意 elisp eval」型 MCP との違い?

単発 eval 型 MCP (例: [[https://github.com/steveyegge/efrit][efrit]]) は哲学的に純粋です: クライアント側ロジックゼロ、
エージェントが全タスクで elisp を書きます。トレードオフは /機能ごとの承認切り分け/:
単一 eval ツールを =:read-only= として登録できないので、読み取りを
自動承認し書き込みだけ確認を要求、という分け方ができません。

anvil の default install は約 40 の named primitive。optional module を enable すると追加ツールが register される を公開し、該当するものは =:read-only=
hint 付きで登録されています。Claude Code の =/permissions= と綺麗に噛み合い、
~org-read-outline~ / ~file-read~ / ~elisp-describe-function~ は自動承認、
~file-batch~ / ~org-edit-body~ / ~file-replace-string~ は確認要求、という
分離が可能です。anvil も ~emacs-eval~ / ~emacs-eval-async~ を escape hatch
として備えていますが、それが 80% のパスではない、という位置づけです。

** …bash から =emacsclient= を呼ぶ方式との違い?

私自身そこから始めました。離れた理由:

- *機能ごとに承認を切り分けられない。* 全 =emacsclient -e '...'= は =/permissions= からは
  "Bash" として一括で見えます。MCP なら tool 単位で =:read-only= hint
  を登録でき、読みは自動承認・書きはプロンプト、の分離ができます。
- *スキーマがない。* エージェントは prompt から elisp API を覚える
  必要があります。MCP は tool list を system schema として公開するので、
  毎回思い出させなくても正しいツールが選ばれます。
- *エスケープ地獄。* bash を通した複数行 elisp は極めて脆く、
  日本語パスや regex を含むと特に悲惨です。
- *構造化された返り値。* MCP tool は typed JSON を返します。
  =emacsclient= は印字された sexp を自力で parse することになります。

中身は同じ Emacs。MCP 層はそこに *型情報と機能ごとの承認* を持たせた入り口、というだけです。

** …org-mcp との違い?

[[https://github.com/laurynas-biveinis/org-mcp][Org-mcp]] は MCP 上の org 編集 primitives に絞って強い package です。
anvil はその primitives (拡張 + bug fix 込み) を同梱しつつ、
file / subprocess / heavy-op / worker-pool レイヤを足しているので、
単一の MCP サーバで org に限らない作業フロー全体をカバーします。
org-mcp 単体で満足している場合、anvil は置換品ではなく /org-mcp の
発想を file/subprocess/worker まで広げたもの/ とお考えください。

** …Emacs を一度も触ったことが無いんだけど?

問題ありません。Anvil はバックグラウンド daemon として動く MCP server
で、あなたが Emacs の UI を開く必要は一切ありません。普段使っている
エディタ (VSCode / Cursor / Neovim / Claude Code / Codex CLI 単体 等) はそのまま
で OK です。

実際に必要なのは:

- Emacs を 1 度インストールして =emacs --version= で確認
- =--fg-daemon= で常駐させる (起動スクリプトはリポジトリに同梱)
- AI クライアントの MCP 設定に 1 ブロック追加

詳しい手順は [[https://github.com/zawatton/anvil.el/blob/master/README.ja.org#emacs-ユーザーじゃない方へ--トークン削減目的で導入する場合][Emacs ユーザーじゃない方へ]] 節にまとめてあります。
Emacs Lisp の知識は不要、AI のトークン消費削減だけを得られます。

* これからの方向性に貢献するには

この README のどこかで刺さるものがあったら是非こちらをお願いします。
私にとってどれも大変嬉しく思います。

- リポジトリへの *Star* ── どんな文脈で響いたのか、反響の波として見えます。
- あなたが実際に使っている org / MCP のワークフローを書いた *Issue* ──
  次に作るべきプリミティブは、この声で決まります。
- [[https://github.com/laurynas-biveinis/org-mcp][org-mcp]] や、私が気付かずに再発明してしまった既存 MCP ラッパーとの、率直な比較。
- ツールの命名と粒度へのフィードバック ──
  「小さい op が多すぎる」のか「これくらいが丁度いい」のか、感覚を教えてもらえると助かります。
