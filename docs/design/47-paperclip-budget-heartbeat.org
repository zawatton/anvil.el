~PHASE-0-2A-SHIPPED~ (2026-05-25) — Paperclip (動画 yt_20260524_paperclip_multi_agent_
management) の二重安全装置 (ハートビート + 予算ハードストップ) を
anvil-orchestrator + anvil-worker-probe に取り込む設計案。動画の
「月 30 万円超え暴走事故」を防ぐ実用機能。

実装状況 (2026-05-25):

- *SHIPPED* — Phase 0 manual emergency stop。公開 API
  =anvil-orchestrator-stop= と MCP tool =orchestrator-stop= を追加。
  全 queued / running task を一括で既存 =anvil-orchestrator-cancel= 経路に
  流し、queued は即 cancelled、live subprocess は SIGTERM → 2 秒後
  SIGKILL の既存 grace を使う。done / failed / cancelled は変更しない。
  これは Paperclip 型 hard stop の手動 kill switch で、budget/heartbeat
  自動発火の前段として使える。ERT 追加。
- *SHIPPED* — Phase 1a budget hard-stop guard。provider の既存
  =cost-estimator= を使い、見積が task budget を超える queued task は
  spawn 前に failed にする。任意 defcustom
  =anvil-orchestrator-daily-budget-usd-total= と
  =anvil-orchestrator-daily-provider-budget-usd= を追加し、設定時は当日
  実費 + running 予約 + 起動予定 reservation が cap を超える task を
  spawn 前に止める。finalize 後に cap 到達した場合は残 queued/running
  task を自動 cancel。
- *SHIPPED* — Phase 2a heartbeat timeout。process filter が stdout 最終
  出力時刻を更新し、pump の timeout scan が =:heartbeat-timeout-sec= または
  =anvil-orchestrator-heartbeat-timeout-sec-default= を超えた無出力 running
  task を =heartbeat-timeout= として SIGTERM → SIGKILL grace に流す。
  default は nil で、静かな長時間 task を誤停止しない opt-in。

* Motivation

anvil-orchestrator は codex / claude / gemini 等を dispatch するが:

- per-task cost 上限なし → 暴走で高額請求リスク
- per-provider cost 上限なし → 1 provider に偏る
- 90%/100% 警告 → 強制停止のフローなし
- ハートビート機構はあるが orchestrator 側で能動的に使っていない

動画の実例:
- 月 30 万円超えの暴走請求
- 「実行中のまま 1 時間放置」問題

memory `feedback_codex_chatgpt_usage_limit` でも触れた通り、codex 経路
は ChatGPT 課金で per-token cost が見えにくいが、API 経路は直接効く。
budget hard stop は **API 経路で必須**。

* Open Questions (要議論)

** Q1. 予算単位の階層

Paperclip は 4 層 (会社/チーム/エージェント/タスク)。1 人事業主の anvil
では:

- (a) per-task のみ (シンプル、Paperclip 最小単位)
- (b) per-task + per-provider 2 層 (provider 別月額上限)
- (c) per-task + per-provider + per-day 累計 (運用しやすい)
- (d) 4 層フル (over-engineering)

→ **推奨**: (c)。task ハードストップ + provider 月額上限 + 日次累計
   ガード。3 層で運用負荷低く、99% の事故を防げる。

** Q2. cost 計測の精度

provider 別に cost 取得方法が異なる:

| Provider     | cost 取得方法                                    |
|--------------+-------------------------------------------------|
| Claude API   | response.usage の input/output tokens × 単価    |
| codex CLI    | =/cost= command output (現在は per-session のみ) |
| gemini API   | response.usageMetadata                          |
| ChatGPT Plus | 取れない (Pro plan)                              |
| Codex Plus   | 取れない (Plus plan)                             |

ChatGPT/Codex Plus 経路は per-token cost が 0 (= サブスク内) なので、
budget は **API key 経路のみ実効**。Plus 経路は wall-clock + 回数だけ
で管理する。

→ budget 機能の defcustom で「API only / both」を選べる。default は
   API only (Plus 経路は無効化、wall-clock で別管理)。

** Q3. ハートビート interval と timeout

論文 (Paperclip 動画) では「N 秒ごとの生存信号」「無応答で自動強制停止」。
anvil-worker-probe quick-alive-p は既に存在するが orchestrator 側で
能動的にポーリングしていない。

選択肢:

- (a) anvil-orchestrator が 30s ごとに probe 呼び出し、timeout 90s
- (b) worker 側が能動的に anvil-state に "alive at <ts>" を書く、
      orchestrator が読み取り
- (c) 両方 (冗長だが安全)

→ **推奨**: (a)。既存 probe を再利用、orchestrator 内で interval/timeout
   を defcustom 化。

** Q4. 強制停止後の cleanup

stop trigger 発火後:

- worktree の uncommitted changes をどうするか
- 中途 commit を残すか revert するか
- worklog にどう記録するか
- next session で resume できるよう snapshot するか (anvil-session 連携)

→ **推奨**: snapshot は anvil-session に委譲、worktree は **そのまま残す**
   (uncommitted changes は user judgment)、worklog に "stopped by budget"
   タグで記録。

** Q5. 90% 警告の通知経路

Paperclip では「90% 警告」を出す。anvil 文脈:

- (a) emacsclient で minibuffer message
- (b) anvil-session-events に emit、Claude が次ターンに気づく
- (c) Stop hook で additionalContext 注入 (anvil-compact 流儀)
- (d) システムレベル通知 (notify-send / Windows toast)

→ **推奨**: (b) + (c)。Claude 自身が「あと 10% 残ってる、ペース落とそう」
   と判断できる経路と、emergency 用の external 通知。

** Q6. budget reset cycle

- per-task: タスク終了で reset (自明)
- per-provider 月額: 月初 reset (anvil-cron で 1 日 0:00 に)
- per-day: 日次 reset (anvil-cron で 0:00)

reset event を anvil-worklog に "budget rollover" として記録、トレース
可能に。

** Q7. Paperclip 全体取り込み vs 部分

| 機能                              | 採用 | 理由                              |

* 評価メトリクス

- 暴走防止: budget 超過時の停止までの latency (target ≤ 5s)
- 誤検知: 正常タスクが budget 内で停止される false positive (target 0)
- ハートビート: stale detect から stop までの latency (target ≤ 120s)
- per-task cost 精度: cost-meter と provider 請求書の乖離 (target ≤ 5%)

* 安全策

- budget 0 設定で全停止 (kill switch)
- stop-trigger 失敗時の二重 kill (SIGTERM → SIGKILL)
- emergency stop MCP tool は authentication 不要 (最速で止める)
- budget reset は monotonic clock で計測 (clock skew 防止)
- Plus plan 経路の budget は wall-clock based、誤って 0 になっても停止
  しない (= 課金されないので無害)

* 取り込み見送り判断材料

- Phase 1 で API 経路の cost-meter 精度が 5% 以内に収まる
- Phase 2 で stale detect の false positive rate が < 1%
- 1 回の暴走を実証的に防ぐ (synthetic 暴走テスト)
