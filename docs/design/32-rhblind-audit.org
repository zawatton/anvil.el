* 設計思想

**pure-elisp server** を売りにしている:

- 外部ブリッジを挟まず Emacs 内に JSON-RPC 2.0 server を立てる
- =socat= は MCP client (stdio) → socket のためだけの薄いラッパー
- =mcp-server.el= + =mcp-server-transport-unix.el= + =mcp-server-transport-tcp.el=
  (TCP は planned stub) + =mcp-server-security.el= + =mcp-server-tools.el=
  のモジュラー構成

**security-first**:

- =mcp-server-security-dangerous-functions= に ~40 関数の blocklist
  (=eval=, =shell-command=, =delete-file= 等)
- =mcp-server-safe-eval= が form を再帰 traverse → blocklist hit で reject
- minibuffer 経由の "(y)es, (n)o, (!)always" permission prompt
- 1000 件 rotating audit log
- sensitive file pattern + sensitive buffer pattern

**認めている制約** (README から直接):

- let-binding 位置の静的解析不完全
- =intern= 経由の dynamic function 呼び出しで blocklist 回避可能
- macro expansion 前の form だけ解析 — 内側が見えない

* 脅威 assessment

**market overlap**: *低い* — 想定 user 像が違う。

- rhblind: 「安全に elisp を LLM に投げたい」「既存 Emacs の diagnostics が
  欲しい」ライト user 向け。pure-elisp で Windows 非対応、tool 2 本に
  絞ることが *feature* (攻撃面 minimization)。
- anvil: 「LLM に Emacs を workbench として使わせる」heavy user 向け。
  org / git / browser / orchestrator まで広げる "workbench MCP server"。

**star 差**: rhblind 68 / anvil 現況 (要実測)。competitor_uptake の theeseuus
sampling で rhblind は r/emacs 界隈で認知されているが、市場を奪い合う対象
ではなく「異なる niche」。

**リスク**: 記号 AI 文脈で "pure-elisp MCP" narrative が強く浸透すると、
「anvil は bash wrapper 挟んでる (= pure ではない)」批判が来る可能性。
ただ実利で言えば bash wrapper は Windows 互換の reason で意図的選択。

**機会**: rhblind の security approach は anvil が軽視してきた領域。
allowlist 側をちゃんと整備する reference にできる (ただし anvil は
eval 系を既に narrow に絞っている前提あり)。

** P1 (検討)

1. *README.org に positioning 比較 section を 1 段落追加*
   - 「anvil は workbench 志向、rhblind/emacs-mcp-server は secure eval
     gateway 志向、用途が違う」と明示
   - theeseuus discussion で「何で anvil 選ぶ?」と聞かれた時の答えに
     なる。書いておくと判断時間を節約できる
   - 工数: 15 min

2. *anvil-security.el という軽量 module を検討* (未実装、draft のみ)
   - =anvil-get-allowed-files= に allowed-tool / read-only-mode /
     dangerous-function blocklist を足す
   - ただし anvil の多くの tool は既に *特定操作* に絞っているので
     rhblind 型の generic elisp eval 防衛は不要
   - 価値: 疑り深い user への disclosure material
   - 工数: design 30 min、impl 2-3h (Phase 分割必要)

3. *rhblind の audit log pattern 採用検討*
   - 1000 件 rotating audit log は単純で有用
   - anvil-state に appendix log として乗せられる
   - 工数: 30 min

** P2 (棚上げ)

- TCP transport — rhblind は planned。anvil は stdio bridge で十分、
  TCP への需要が来るまで保留。anvil-pty-broker が既に TCP 化 pattern
  を持っているので必要になれば参照できる。
- permission prompt — anvil は headless (Claude Code 経由) 前提、
  minibuffer prompt は合わない。
