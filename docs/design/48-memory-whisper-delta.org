* Motivation

** 現に起きている問題 (active bug)

SessionStart の system reminder が現在こう警告している:

#+begin_quote
WARNING: MEMORY.md is 34.5KB (limit: 24.4KB) — index entries are too
long. Only part of it was loaded. Lines after 200 will be truncated.
#+end_quote

Doc 29 の =memory-regenerate-index= は MEMORY.md を **decay-score 降順の
pointer list 全量** で生成し、SessionStart で全 load する設計
([[file:29-memory-engine.org]] 本文 L133)。memory 件数が増えるほど
MEMORY.md が肥大し、ある時点で **末尾エントリが切り捨てられて
load されなくなる**。つまり古い・低 decay の memory は事実上
SessionStart の文脈から消える。

Doc 37 の auto-inject は別経路で =max 1000 tokens=・優先度
=recency × importance × project_match= の capped 注入を既に持つ
([[file:37-claude-mem-bridge.org]] L184)。が、これは observation memo
(claude-mem 由来 capture) 向けであって、Doc 29 の MEMORY.md pointer
index 全量 load 問題そのものは解消していない。

** WHISPER が効く理由

毎セッション MEMORY.md を全量 load する代わりに:

1. SessionStart は「前回セッション終了時点の index snapshot」と
   現在の index を比較し、*差分 (added / changed / removed)* だけを
   注入
2. 変化が無ければ何も注入しない (= 大半のセッションはほぼ 0 トークン)
3. 全量が要るとき (新規 PC / snapshot 不在 / 明示要求) だけ full load
   に fallback

これで MEMORY.md の物理肥大と SessionStart の token 負荷を恒久的に
切り離す。truncation bug も「全量を毎回送らない」ことで自然に解消。

* Open Questions (要議論)

** Q1. 「差分」の単位と粒度

候補:

- (a) MEMORY.md の *行* 単位 diff (= Subconscious WHISPER 素朴版)
- (b) memory *エントリ* 単位 (name 単位で added/changed/removed)
- (c) エントリ単位 + 本文 hash で「説明文だけ変わった」も検出

→ *推奨*: (b) エントリ単位。MEMORY.md は 1 entry = 1 行なので行 diff
   と近いが、entry の =name= を安定キーにすれば decay-score の並び替え
   だけで「changed」誤検出しない。本文 (description) hash も持てば
   (c) に拡張可。

** Q2. snapshot をどこに持つか

last-session の index 状態を保存する場所:

- (a) =anvil-state= に =(:memory-index-digest <sha1> :entries [...])=
- (b) memory DB に専用テーブル =memory_session_snapshot=
- (c) =~/.claude/.../memory/.last-index.json=

→ *推奨*: (a) anvil-state。session スコープ管理は anvil-state の責務
   ([[file:08-state.org]] 系)。cross-PC は Doc 29 と同じ git-tracked DB の
   仕組みに乗らないので、snapshot は **PC ローカル** で良い (差分が
   取れない PC では full fallback すればよい)。

** Q3. full-load fallback の発火条件

差分注入が成立しないケース:

- snapshot が存在しない (初回 / 新規 PC / state クリア後)
- snapshot が古すぎる (N セッション以上前 / TTL 超過)
- 差分が index の X% を超える (= 実質作り直し、差分の方が高コスト)
- ユーザーが明示 (=ANVIL_MEMORY_INJECT=full=)

→ *推奨*: 上記 4 条件のいずれかで full load。閾値は
   defcustom (=anvil-memory-whisper-full-threshold= 既定 0.4)。

** Q4. Doc 37 auto-inject との責務分担

Doc 37 は既に capped/prioritized 注入を持つ。二重注入を避ける:

- Doc 29 MEMORY.md = *構造化 memory* (user/feedback/project/...) の
  index → WHISPER delta はこちらを担当
- Doc 37 observation memo = *作業 capture* (claude-mem 由来) → 既存
  capped 注入のまま

→ *推奨*: 経路を分離。WHISPER は Doc 29 系 index 専用。Doc 37 経路は
   無改変。両者の合計 token を =memory-cost= 的 MCP tool で可視化。

** Q5. Sleep-time consolidation (Stop hook 非同期) を入れるか

Subconscious は Stop hook で transcript を送って記憶整理。anvil 文脈:

- (a) 入れない。memory-add は明示 tool 呼び出しのまま (現状維持)
- (b) Stop hook で worktree の memory dir を =memory-scan= + index
  snapshot 更新だけ非同期実行 (記憶「生成」はしない、snapshot を
  次回 delta の基準に固定するため)
- (c) (b) + orchestrator で「今回 session の学びを memory 候補に」要約

→ *推奨*: まず (b)。WHISPER の差分基準 snapshot を Stop 時に確定させる
   のが主目的。(c) の自動 memory 生成は誤検出リスクが高く、既存の
   =memory-save-check= 重複検出を通さず増殖する恐れがあるので Phase を
   分ける。=child.unref= 相当は Doc 36 (auto-compact) の非同期 snapshot
   パターンを流用。

** Q6. 切り捨てられている既存 entry の救済

現状 MEMORY.md は 200 行で切れ、末尾 entry が load されていない。
WHISPER 化しても「過去に一度も注入されていない低 decay entry」は
依然 cold のまま。

→ *推奨*: 初回 full load 時に **全 entry を 1 度は注入**して snapshot
   を確定。以降は delta。これで「切り捨てゾーンの entry」も最低 1 回は
   文脈に入る。あわせて MEMORY.md index 自体の 1 行長を縮める
   (description を短縮、Doc 29 の警告どおり ~150 字以内) lint も同時に
   回す。

* 評価メトリクス

- SessionStart で注入される memory token (target: 変化なしセッションで ≈ 0)
- full-load 発火率 (target: 定常運用で < 10% のセッション)
- 切り捨てゾーン entry が「最低 1 回は注入された」率 (target: 100%)
- delta 誤検出率 (decay 並び替えだけで changed と誤判定する率、target: 0)

* 安全策

- snapshot 破損 / 不整合時は **必ず full load に fallback** (memory が
  見えなくなる事故を最優先で防ぐ)。
- delta が「全 entry removed」を示したら異常とみなし full load
  (state クリア誤爆対策)。
- Stop hook の非同期処理は session 終了をブロックしない
  (=child.unref= 相当)。失敗しても次回 full fallback で復旧。
- snapshot は PC ローカル。cross-PC では差分が無いので full から開始
  (正しい挙動)。

* 取り込み見送り判断材料

- Phase 6a で「変化なしセッションの注入 token ≈ 0」かつ memory の
  実利用 (memory-access 経由参照) が full load 時と同等以上を保つか
- full-load 発火率が定常で < 10% に収まるか (毎回 full なら無意味)
- 切り捨てゾーン entry の cold 化が解消するか

いずれも未達なら Doc 29 の「MEMORY.md 1 行短縮 + 件数 prune」
([[file:29-memory-engine.org]] decay GC) だけで足り、WHISPER は過剰と判断。
