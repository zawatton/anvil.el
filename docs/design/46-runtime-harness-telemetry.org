** D1. classifier rule の初版

dispatcher 内 static 同期分類、LLM fallback なし。90%+ は regex + error
type で判定可能との前提。LLM 分類は Phase 4 で opt-in。

#+begin_src elisp
(defconst anvil-harness-telemetry-classifier-rules
  '(;; no-exec (実行不能 ~ 33.3%)
    (no-exec
     :error-symbols (void-function
                     error-process-exited-abnormally
                     file-missing
                     file-error)
     :regex ("not callable"
             "Connection refused"
             "No such file or directory"
             "Permission denied"
             "process exited [1-9]"))

    ;; contract-violation (契約違反 ~ 23.3%)
    (contract-violation
     :error-symbols (wrong-type-argument
                     wrong-number-of-arguments)
     :regex ("Missing required parameter"
             "json-schema validation"
             "Invalid argument"
             "InputValidationError"))

    ;; stall (反復停滞 ~ 13.2%) — repetition-detector emit のみ
    (stall :emit-only-from-repetition-detector t)

    ;; reasoning (推論ミス ~ 30%) — fallback
    (reasoning :fallback t)))
#+end_src

注意: =wrong-type-argument= は call stack depth で意味が変わる
(dispatcher 入口 = contract-violation、内部 = no-exec)。Phase 1
classifier は **stack depth で分岐** する。

** D2. event 書込み先 = 同 DB 新 table =harness_failures=

anvil-worklog 本体は人間の作業ログ、failure event は機械生成で性格が違う。
DB file は共有 (=Notes/.anvil-worklog/anvil-worklog-index.db=) するが
table は独立、FTS5 も別 index。

#+begin_src sql
CREATE TABLE harness_failures (
  id INTEGER PRIMARY KEY,
  timestamp TEXT NOT NULL,        -- ISO datetime
  class TEXT NOT NULL,            -- no-exec/contract-violation/stall/reasoning
  confidence REAL NOT NULL,       -- 0.0-1.0 (D4)
  tool TEXT,
  args_digest TEXT,               -- sxhash for repetition-detector
  error_message TEXT,
  provider TEXT,                  -- claude/codex/gemini/etc
  session_id TEXT,
  machine TEXT,
  raw_context TEXT                -- 短縮された原文
);

CREATE VIRTUAL TABLE harness_failures_fts USING fts5(
  error_message, raw_context,
  content='harness_failures', content_rowid='id',
  tokenize='trigram'
);
#+end_src

* Motivation

anvil-orchestrator / anvil-worker / anvil-claude-watchdog 等で
失敗 event は発生しているが、現状は:

- error message を raw string で worklog に書く
- recovery ロジックは module ごとに ad-hoc
- 改善 PR の効果を **何が減ったか** で測れない

論文の 4 分類で telemetry を分類すれば:

1. **改善 PR の効果が定量化できる** (例: Tool Attention は契約違反を
   減らすはず、TACO は反復停滞を減らすはず)
2. ボトルネック特定: 自分の運用では実行不能 33% が正しいのか?
3. 後続 doc (43/44/45/47) の取り込み効果を逆算で評価可能
4. anvil の外部公開時に「論文準拠の telemetry」として差別化

これは「単独で効果薄、後続改善の基盤」という性格上、**最優先で着地**
させる価値がある。

* Open Questions (要議論)

** Q1. 分類器をどこに置くか

選択肢:

- (a) dispatcher の error handler 内で同期分類 (低レイテンシ、低精度)
- (b) anvil-worklog 書込み時に post-process で分類 (中レイテンシ、中精度)
- (c) LLM を使って分類 (高精度、cost あり)

→ **推奨**: (a) static rules を default、(c) を opt-in。論文の 4 分類は
   regex + error type で 90%+ 判定可能。LLM 分類は曖昧ケースのみ。

** Q2. 後続 doc 効果測定の baseline をどう取るか

各 doc を ship する前に baseline 計測が要る:

- Tool Attention (Doc 44) ship 前後で「契約違反」が減るか
- TACO (Doc 45) ship 前後で「反復停滞」が減るか
- autoresearch (Doc 43) ship 前後で「推論ミス」が減るか
- Paperclip (Doc 47) ship 前後で「実行不能」 (= stale 含む) が減るか

→ telemetry MUST ship 最初に。それまで Doc 43-47 の効果は逸話的にしか
   語れない。

** Q3. プライバシー / 漏洩リスク

failure event の中に user data (file path, prompt 抜粋) が含まれる:

- ローカルにのみ保存 (= anvil-worklog の既存挙動と同じ)
- 集計値のみ外部公開可能 (count by class, per-day)
- raw event は disk only、export には opt-in

→ 既存 worklog ポリシーで十分。

** Q4. 統計 dashboard をどう見せるか

- (a) MCP tool =harness-telemetry-stats= で JSON 返す (Claude 用)
- (b) Emacs tabulated-list-mode で見る (人間用)
- (c) HTML 生成 (anthropic_markdown_to_html 流儀、外部共有用)

→ (a) を Phase 1、(b) は anvil-compact Phase 3b のパターン流用。
   (c) は marginal。

** Q5. 反復停滞の閾値設定

論文では 4 パターン (同一思考反復 / 状態変化なし / 進行なし / トライ飽和)。
anvil での閾値:

- 同一 tool+args 3 回連続 → 停滞警告
- 5 回連続 → 停滞分類 + 段階介入トリガ
- session-events 連続 review 5 件 → 停滞分類

defcustom で調整可能に。default は論文寄りの保守的値。

** 主要 component

#+begin_example
[anvil-harness-telemetry]

* 安全策

- classifier 失敗時は raw error をそのまま記録 (フォールバック)
- worklog 書込みエラーは silent fail (telemetry が main flow を止めない)
- repetition-detector の ring buffer overflow 防止 (default size 100)

- 1 失敗 event 記録コストが orchestrator 通常使用の 1% 以内
