Tier 3 of Week-2 roadmap.  Ephemeral-state persistence that sits
between a single Claude conversation and the long-term memory system.

* Motivation

Today's memory system stores durable facts (user preferences, project
patterns, known incidents).  A Claude conversation also builds up a
second class of state that *dies when the session ends*:

- Current branch + base branch of active work.
- Baseline test results captured at session start.
- Open/edited buffers relevant to the task.
- Intermediate reasoning: "I ruled out X because Y".
- Task list progress.

Memory is the wrong place for this — too noisy, expires naturally.
Git branch + worktree state encode some of it.  What's missing is a
*structured session record* that a resuming Claude (same topic, new
session) can read in 1 call and pick up from.

Doc 17 fills this gap with a thin SQLite-backed rec.  Think of it as
"bookmarks for Claude".

* Goals

1. ~session-snapshot NAME~ captures active state into a named rec.
2. ~session-resume NAME~ returns that rec, along with a suggested
   preamble string.
3. Snapshots survive daemon restart (persisted via anvil-state, ns
   =session=).
4. Auto-expire after TTL (default 14 days).

* Non-goals

- Full replay (tool call history replay is not safe).
- Cross-machine sync (single-machine anvil-state; user's Git is
  synchronization).

* Architecture

#+begin_src elisp
(:name STR :created-at TIMESTAMP
 :branch STR :base-branch STR
 :buffers (list (:file PATH :line N :last-edit TIMESTAMP))
 :baseline-tests (:passed N :total N :failed N)
 :task-summary STR        ; human/Claude written
 :notes LIST              ; bulleted reasoning
 :preamble-suggested STR)
#+end_src

Captured automatically via helpers (read git branch + open
buffer-list + run test-baseline); caller supplies :task-summary and
:notes.  Restored: emit a formatted string suitable for
~orchestrator-preamble-set~ so the resuming session gets context
without a wall of Read calls.

* Public API

#+begin_src elisp
(anvil-session-snapshot NAME &key task-summary notes)
  ;; Captures current state, stores, returns the plist.

(anvil-session-resume NAME)
  ;; Returns the plist (or nil).  Does not mutate current state.

(anvil-session-list)
  ;; Returns list of (:name :created-at :branch :task-summary-head).

(anvil-session-delete NAME)
  ;; Purge one snapshot.
#+end_src

*** Hook 一覧

| Hook             | 発火タイミング                    | Role                              |
|------------------+-----------------------------------+-----------------------------------|
| =SessionStart=   | 会話開始時 / =--continue= 時      | 前回 snapshot を復元              |
| =PreCompact=     | context window 圧迫で compact 直前 | working state を snapshot         |
| =PostToolUse=    | 各 tool call 後                   | event を FTS5 に index            |
| =UserPromptSubmit= | ユーザー入力確定時              | 意思決定 / 訂正を capture         |
| =SessionEnd=     | /exit 時                          | 最終 snapshot + TTL reset         |

*** 実装戦略

Claude Code の hook は shell command を発火するだけなので、各 hook
から anvil MCP を叩く thin wrapper を用意する:

#+begin_src bash
# ~/.claude/settings.json (hooks section)
{
  "hooks": {
    "PreCompact": "anvil-hook pre-compact $CLAUDE_SESSION_ID",
    "SessionStart": "anvil-hook session-start $CLAUDE_SESSION_ID",
    "PostToolUse": "anvil-hook post-tool-use $CLAUDE_TOOL_NAME",
    "UserPromptSubmit": "anvil-hook user-prompt $CLAUDE_PROMPT"
  }
}
#+end_src

=anvil-hook= は POSIX shell script、=emacsclient -e
'(anvil-session-hook-dispatch ...)'= に forward。Emacs server 側で:

#+begin_src elisp
(defun anvil-session-hook-dispatch (event &rest args)
  "Dispatch Claude Code lifecycle hook."
  (pcase event
    ('pre-compact
     (anvil-session-snapshot
       (anvil-session--auto-name)
       :task-summary (anvil-session--derive-task-summary)
       :buffers (anvil-session--capture-buffers)))
    ('session-start
     (let ((last (anvil-session--most-recent-snapshot)))
       (when last (anvil-session--emit-preamble last))))
    ('post-tool-use
     (anvil-session--index-event (car args)))
    ('user-prompt
     (anvil-session--capture-decision (car args)))))
#+end_src

*** Event indexing (FTS5)

PostToolUse / UserPromptSubmit は snapshot とは別に =anvil-session-
events.db= (sqlite FTS5) に per-session log:

#+begin_src
CREATE TABLE events (
  session_id TEXT,
  seq INTEGER,
  ts DATETIME,
  kind TEXT,        -- tool-use / user-prompt / error / decision
  tool TEXT,
  summary TEXT      -- 200-char digest (not full payload)
);
CREATE VIRTUAL TABLE events_fts USING fts5(summary, content='events');
#+end_src

=session-events-search= MCP tool で "前回あの bug を直した時どの
tool 使った?" が検索できる。

*** Context-mode との差分

context-mode は PreCompact で *全 event を compact payload に再
注入* するが、anvil は FTS5 に置くだけで、resume 時も必要な event
だけ検索させる。"don't dump it back, index it" 方針。

*** Preamble の自動生成

SessionStart hook で前回 snapshot があれば、=orchestrator-preamble-
set= に:

#+begin_example
Resuming session "refactor-anvil-worker" (last active 2026-04-20 10:30).
Branch: feat/worker-v2 (base develop).
Last task: "Phase 4a backoff implementation".
Baseline tests: 508/510 passing (2 known flaky).
Recent key events (last 5 tool calls):
  - file-batch: anvil-worker.el (12 edits)
  - ert-run: worker-retry-test (passed)
  - ...
#+end_example

を emit。Claude は 1 回の preamble 読みで context を再構築できる。

** Phase 4 — hook 無効化 / opt-out (~SHIPPED~ 2026-04-22)

=anvil-session-hooks-enabled= defcustom (default nil) で明示 opt-in。
=anvil-hook-install-settings= command が =~/.claude/settings.json= に
hook 5 件を deep-merge する。

ship 差分:
- =anvil-hook-install-settings= の API は keyword-style (=:dry-run=
  / =:uninstall= / =:path= / =:script=) に確定。戻り値 =(:path
  :diff :applied)= で diff 文字列を CLI で見せ、実書込みは applied
  フラグで確認できる。
- =scripts/anvil-hook= (POSIX sh) と =scripts/anvil-hook.bat=
  (Windows cmd.exe) を同梱。POSIX 側は =session-start= の quoted
  string return を unwrap + unescape して、hook stdout が elisp
  文字列構文ではなく素の preamble テキストになるよう調整。
- install-settings は deep-merge: 既存 =hooks.*= のうち anvil 管理
  外の key (user の =PreToolUse= 等) は touch せず、top-level の
  非 hooks key (=model= / =permissions= 等) も全て verbatim。
- =anvil-session-hook-script= / =-claude-settings-path= defcustom で
  マシンごとの path をコードいじらず設定可能。

* Out of scope / future

* Platform notes

- Hook shell command は Windows native では =cmd.exe= 経由になる。
  =anvil-hook.bat= wrapper を別途用意する (Phase 3 実装時)。
- =emacsclient -e= の quoting は OS ごとに差がある。MSYS path /
  Windows path の混在に注意 (Memory
  ~reference_emacsclient_main_daemon_via_mcp_worker.md~ 参照)。
- Claude Code の hook spec (env var 名、timeout) は v1.x で
  stable だが、将来変更あれば wrapper 側で吸収。

* Review Questions

1. Q. Hook の event index を =anvil-session-events.db= に分けるか
   =anvil-state= に namespace で入れるか? → 分ける。FTS5 + 大量
   row の access pattern が =anvil-state= (軽量 KV) と合わない。
2. Q. PostToolUse を全 tool で発火すると event 数が爆発しない?
   → per-tool の =summary= だけで 200 chars 以内に抑える。1 session
   100-500 event 想定で FTS5 なら軽量。
3. Q. SessionStart で preamble を *自動 inject* するか、suggest
   だけか? → suggest (orchestrator-preamble-set は opt-in)。強制
   injection は context の主導権を奪う。
4. Q. 複数 Claude Code instance (並列起動) で session-id 衝突は?
   → Claude Code の =$CLAUDE_SESSION_ID= は uuid、衝突しない想定。
   不安なら hook 内で pid もタグる。
