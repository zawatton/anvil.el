* Motivation

codebase-memory-mcp の核心主張は「coding agent が repo を file
ごとに grep/read して token を溶かす」問題を、コードベース全体を
一度グラフ化し構造クエリ 1 発で置換する点にある (公称: 5 構造
クエリが 3,400 tokens、従来 grep/read 連発で 412,000 tokens)。
これは anvil の token-savior 思想 (Doc 26 manifest / Doc 28
progressive disclosure / Doc 29 memory) と完全に同一線上にある。

anvil 自身が =anvil*.el= + =dev/nelisp/= の巨大 codebase であり、
Claude が毎セッション「この関数を誰が呼ぶか」「この helper は
どこで定義されるか」「この変更の波及範囲は」を grep で何度も
辿っている。call-graph をあらかじめ SQLite に持てば、この hop が
構造クエリ 1 発に畳まれる。

* Goals

1. =defs-index= の SQLite schema に call-edge テーブル (=CALLS=) を
   追加し、定義済みシンボル間の呼び出し関係を永続化する。
2. =trace_path= MCP tool — あるシンボルを起点に BFS で「呼び出し元
   (callers)」「呼び出し先 (callees)」を depth 指定で辿る。
3. =detect_changes= MCP tool — git diff の変更行を defs-index に
   マップし、変更シンボル + その逆参照 (blast-radius) を返す。
4. =dead-code= 検出 — caller 数 0 の非 public シンボルを列挙
   (=defs-index= の degree フィルタで実現)。
5. 既存 =defs-search= / =defs-references= / =defs-signature= /
   =defs-who-requires= は無改変。call-graph は opt-in の追加表。

* Non-goals

- *Cypher query engine* — read-only openCypher サブセットの再実装は
  ROI が低い。実用クエリは SQLite =WITH RECURSIVE= CTE で代替する。
- *Louvain / community detection / architecture clustering* —
  /当初見送り → Phase 5 で実装/。純 elisp の single-level Louvain を
  *ファイル単位*の call graph (数百ノード) に適用 (=defs-clusters= /
  =defs-architecture=)。symbol 単位 (O(10k) ノード) は elisp で重すぎる
  ため file 単位に限定。=get_architecture= 相当を提供する。
- *埋め込みベース semantic similarity (=SIMILAR_TO=)* — nomic-embed
  を vendor する codebase-memory-mcp の方式は native binary 前提。
  semantic 検索は Doc 18 (RAG) の所掌、本 doc では扱わない。
- *HTTP route graph / IaC graph / cross-repo edges* — 事務所 + anvil
  の用途に対し過剰。需要が出るまで保留。

* Architecture (案、議論前提)

** call-edge 抽出
入力ソースは 2 経路:
1. *elisp* — Doc 12 の reader (sexp walk) を再利用。関数本体の
   car が fboundp な symbol を call-site として収集。最も安価で、
   anvil/nelisp 自身に即効。
2. *brace/tree-sitter 言語* — Doc 21 (tree-sitter edits) の AST を
   walk して call expression を収集。Python 先行。
3. fallback — LSP (eglot) の =textDocument/callHierarchy= があれば
   利用、無ければ textual match (同名関数の false edge に注意)。

** schema 追加 (defs-index DB)
=call_edge(caller_id, callee_id, file, line, lang, resolved)= の単一
表。=resolved= は AST/LSP 解決か textual match かのフラグ
(textual は false-positive 可能性を呼び出し側に開示)。

** trace_path
=WITH RECURSIVE= で caller/callee 方向の BFS。depth は 1–5 で
hard cap (codebase-memory-mcp に倣う)。戻りは
(:root SYM :direction callers/callees :depth N :paths (...))。

** detect_changes
=anvil-git= の diff から変更ファイル + 行範囲を取得 → defs-index で
行範囲 → シンボル逆引き → 各シンボルの caller を =trace_path= で
1–2 hop 展開 → risk 分類 (public API か / test 有無 / caller 数)。

* Phase 分割案

** Phase 1 — trace_path over existing refs edges (実装済み 2026-06-20)
当初案 (=call_edge= schema + 新規抽出) は不要と判明: defs-index の
=refs= が既に caller(=context=)→callee(=name=, =kind='call'=) を持つ。
実装は =refs= 上の =WITH RECURSIVE= BFS (=anvil-defs-trace-path=) +
=defs-trace-path= MCP tool + additive index =idx_refs_kind_context=。
=internal-only= (既定 ON) で builtins/macros/third-party を除外し
index 定義済みシンボルに限定。ERT 7 件、anvil.el 自身で dogfood 済。

** Phase 2 — detect_changes (blast-radius) (実装済み 2026-06-21)
=anvil-defs-detect-changes= + =defs-detect-changes= MCP tool。
=git diff -U0 REV= のハンク行範囲 → defs の =[line, end_line]= 重なりで
変更シンボル特定 → caller を =trace_path= で展開 (blast-radius) →
risk 分類 (public / test 参照 / caller 数)。git 依存部と DB 写像部を分離
(=:changes= override seam でテスト可能)、anvil-git は遅延 require。
ERT 4 件、dev/anvil.el 実 diff で dogfood 済。

** Phase 3 — tree-sitter 言語 (Python) (実装済み 2026-06-21)
Doc 21 基盤 (=anvil-treesit-with-root-fn= / =anvil-treesit-compile-query=)
で Python の defs (function/method/class) + call-edge を抽出し、elisp と
同一 =defs= / =refs= に投入。opt-in =anvil-defs-languages= (既定 elisp-only)。
=scan-file= は拡張子 dispatch、既存 elisp 経路は =scan-elisp-file= に改名。
callee は =(call)= の function フィールド (bare / attribute 両対応)、caller
は enclosing function。syntactic 解決 (同名 cross-module は false edge)。
features は空 (require/provide は elisp 概念)。ERT 4 件、実 .py で dogfood 済。

*** Phase 3 JS/TS サブフェーズ (実装済み 2026-06-21)
JavaScript (=.js= / =.jsx=) / TypeScript (=.ts= / =.tsx=) を同型追加。
=anvil-defs--scan-ts-file= が function_declaration / 名前付き arrow・
function-expression binding / class / method を defs、call_expression を
refs に投入。caller は最寄り named function (無名 arrow は variable_declarator
/ pair から命名)、callee は bare / member 両対応。anvil-treesit-backend
のみ依存。ERT 5 件、実 .ts で dogfood。残: Rust / Go 等は同パターンで追加可。

** Phase 4 — dead-code + degree-filter + import graph (実装済み 2026-06-21)
3 点セットで実装:
1. *dead-code* (=defs-dead-code=) — 自己参照を除外した「index 内で
   どこからも参照されない callable 定義」列挙。=:name-pattern= で
   =%--%= 等に絞れる。外部/動的使用は不可視なので *候補* 扱い。
2. *degree-filter* (=defs-callers-rank=) — callable 定義を call in-degree
   (distinct caller 数) でランク。=:min=/=:max=/=:order= で hotspot 抽出
   や低使用 sweep。name で dedup (per-symbol view)。
3. *Py/JS/TS import graph* — Python の import / from-import、JS/TS の
   import specifier を =requires= feature として投入。=defs-who-requires=
   が elisp の require/provide に加え「誰がモジュール M を import するか」を
   多言語で答える。
ERT 6 件 (defs suite 計 42 PASS)、byte-compile clean。実 anvil.el で
dogfood — hotspot top (=anvil-server-with-error-handling= 249 callers 等)、
private dead-code 71 件を検出。

** Phase 5 — clustering / architecture (実装済み 2026-06-21)
見送り群の最初の取り込み。純 elisp の single-level Louvain を
*ファイル単位の call graph* に適用し、モジュール/サブシステムを推定:
1. =defs-clusters= — file 間 call で密結合なファイル群を community 化
   (=:file-pattern= で範囲限定、=:min-size=)。:members は basename。
2. =defs-architecture= — 総数 + hotspots (=callers-rank= 再利用) +
   community 数 + 上位クラスタ を 1 コール。
graph は =refs.file_id= (caller file) → callee def の file の cross-file
edge を重み集計。symbol 単位 (14k ノード) は elisp Louvain で ~590s と
重すぎたため *file 単位* (~500 ノード、graph 264ms + Louvain 321ms) に
確定。Louvain は決定的 (sort 順 + strict-gain move)。
*同時に Phase 4 の =callers-rank= 性能 bug 修正*: per-def 相関サブクエリが
SQLite に full-scan され実 index で ~597s → in-degree を 1 pass 集計して
LEFT JOIN する形に変更し *289ms* に。
ERT 3 件 (defs suite 計 45 PASS)、実 anvil.el で dogfood (architecture
608ms、74 communities、最大クラスタ 86 files)。
