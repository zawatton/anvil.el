* Motivation

anvil ecosystem には改善対象が散在している:

- elisp byte-compile の hot path / inline 判定境界
- NeLisp Phase 47 codegen の SSA pass 順序 / レジスタ割付ヒューリスティック
- MCP tool docstring の品質 (Tool Attention IsO 精度に直結 — Doc 43
  併走候補)
- CLAUDE.md content の構造 (token cost と参照精度のトレードオフ)
- anvil-orchestrator routing rule (provider × task 最適マッピング)
- test-suite ordering (fail-fast 性能)

これらは現状すべて *人が観察 → 仮説 → 1 変更 → 検証* で改善している。
autoresearch の核心は **次の試行を AI が判断する** 点で、従来の AutoML/
Hyperopt が「探索空間を人が設計」していたのに対し、探索対象が *コード
構造そのもの* に拡張される。

anvil で autoresearch を持つと:

1. ベンチマーク駆動の自己改善が 24h 走り続ける (cron + orchestrator
   + worktree の組み合わせで実現可能性は既に揃っている)
2. anvil-bisect の 1 次元探索を多次元化した上位 primitive
3. NeLisp Phase 47 codegen 改善のような *人間の感性が効きにくい*
   領域で AI 独自発見が起きる可能性 (動画事例の非対称 Dropout 同様)

|                                 | recall)          |              |        |

** Q3. 評価関数の独立性 (メタオーバーフィット対策)

原典の最大の罠: ループ内バリデーション集合への過適合。
対策はホールドアウト集合を **ループ外** に持つこと。

anvil 文脈での具体策:

- 評価データを 2 分割: =eval-inner= (ループ内) + =eval-holdout=
  (週 1 回のみ実行)
- =eval-holdout= の score が下がっていたら採用変更を retroactive に
  revert する gate を anvil-autoresearch-finalize に持つ
- holdout は git で snapshot 化して時系列で差分追跡

** Q4. 安全策 (worktree + sandbox + budget)

動画推奨は Docker 分離 + branch 分離。anvil 既存は:

- =anvil-git worktree-list= で worktree 分離は primitive 化済
  (memory: project_anvil_git_extension)
- sandbox は orchestrator codex 経路で =:sandbox "workspace-write"=
- Docker 分離は anvil に primitive なし — 新規追加かパススルー

→ 初手は worktree + codex sandbox で十分、Docker は Phase 2 以降。

ただし以下 4 つの hard gate は MVP 必須:

1. cost budget hard stop ([[file:10-orchestrator.org]] 既存 :budget
   流用)
2. wall-clock timeout (loop 全体 + 1 試行ごと)
3. file scope whitelist (autoresearch が touch できる path を allowlist)
4. eval-holdout 劣化検知時の auto-revert

** Q5. anvil-bisect (1 次元探索) との関係

既存の =anvil-bisect= は test-driven の 1 次元 git bisect。
autoresearch は多次元 + AI 判断 + 永続ループという拡張。

選択肢:

- (a) anvil-bisect を内部で autoresearch が呼ぶ (1 次元探索のサブルーチン)
- (b) 完全独立、bisect は触らない
- (c) bisect の API を拡張して autoresearch も同じ MCP tool 表面に出す

→ **推奨**: (b) 独立。bisect の単純さを壊さない、autoresearch は別 module。

** Q6. orchestrator preset として実装するか、新 module として実装するか

選択肢:

- (a) =orchestrator-submit= の preset =:autoresearch= として実装
- (b) 新規 =anvil-autoresearch= module、内部で orchestrator を呼ぶ

→ **推奨**: (b)。loop state machine + snapshot + holdout gate が
複雑で、preset 1 個に押し込むのは無理。orchestrator は試行 dispatch

** 主要 component

#+begin_example
[anvil-autoresearch]
   ├─ run-loop (cron-friendly entry point)
   │    ├─ propose-change       → orchestrator-submit
   │    │    (codex, :sandbox workspace-write)
   │    ├─ apply-change         → worktree commit
   │    ├─ eval-inner           → anvil-bench / make test

   │    ├─ commit-or-revert     → git worktree

   ├─ eval-holdout              (週 1, 別 cron entry)

   └─ MCP surface (tools)
        ├─ autoresearch-start  / stop / status

#+end_example

** Phase 1a — MCP docstring evaluator (shipped)

- =autoresearch-docstrings= MCP tool を追加
- 登録済み MCP tool manifest を read-only に評価
- score 対象:
  - description が短すぎる
  - generic phrase だけで用途が曖昧
  - usage guidance (=use/when/filter/optional/required/returns 等) が薄い
  - tool id の有意味 token が description に出ていない
  - input schema parameter description が無い/短すぎる
- 返却項目:
  - server-id / id / handler / score / issues
  - description-length / param-count / description
  - suggested-action = rewrite-mcp-description

この段階は evaluator のみ。LLM rewrite、patch 生成、評価データ分割、
holdout gate、keep/discard は Phase 1b 以降。

** Phase 1b — MCP docstring rewrite planner (shipped)

- =autoresearch-docstring-plan= MCP tool を追加
- =tool_id= / =server_id= 指定時は対象 tool の plan を返す
- =tool_id= 省略時は =autoresearch-docstrings= の最高 score 候補を選ぶ
- read-only 返却項目:
  - current description
  - score / issues
  - schema parameter docs
  - registration-site hint (= :id 検索による file/line/snippet)
  - deterministic draft description
  - future rewrite 用 prompt

この段階では patch を作らない。次段は planner の prompt を orchestrator に
渡す、または deterministic draft を使って dry-run patch を生成する。

** Phase 1c — MCP docstring dry-run patch candidate (shipped)

- =autoresearch-docstring-patch= MCP tool を追加
- =tool_id= / =server_id= 指定時は対象 tool の patch candidate を返す
- =tool_id= 省略時は planner と同じく最高 score 候補を使う
- =new_description= 省略時は deterministic draft を replacement に使う
- read-only 返却項目:
  - dry-run marker
  - file / start-line / end-line
  - old/new description
  - old/new Elisp string literal
  - unified-diff style patch candidate
  - 元 planner 結果

ここまでで Q1-A は「候補抽出 → plan → dry-run patch」まで到達。次段は
patch candidate を apply する前の validation gate、または orchestrator 経由
の LLM rewrite を dry-run に接続する。

** Phase 1d — MCP docstring local apply/eval loop (shipped)

- =autoresearch-docstring-apply= MCP tool を追加
- Phase 1c の patch candidate を土台に、対象 =:description= string literal
  の stale bounds を検証して 1 箇所だけ書き換える
- =eval_command= が指定されていれば shell command を実行し、exit 0 なら keep
- =eval_command= が non-zero exit の時は、元ファイル content を復元して
  =:reverted t= を返す
- =keep_on_eval_failure= が truthy の時だけ、失敗した変更を残す
- LLM 呼び出し・worktree 分離・budget 管理はまだ行わない

これで Q1-A は「候補抽出 → plan → dry-run patch → apply → eval →
keep/revert」まで到達。ただし deterministic draft / user-provided rewrite
専用で、orchestrator codex 経由の proposal generation は Phase 1e へ送る。

** Phase 1e — MCP docstring orchestrator proposal submit (shipped)

- =autoresearch-docstring-propose= MCP tool を追加
- Phase 1b の plan から、replacement description だけを返すよう指示した
  Codex/orchestrator 向け prompt を生成する
- =anvil-orchestrator-submit= に 1 task を async 投入する
- task には以下を明示的に載せる:
  - =:budget-usd=
  - =:timeout-sec=
  - =:heartbeat-timeout-sec=
  - =:cwd= (repo/worktree directory)
  - =:sandbox= (Codex sandbox override)
- 戻り値:
  - =:batch-id=
  - 実際に submit した =:task=
  - 元 plan

この段階でも patch は適用しない。proposal 結果を review し、採用する場合は
=autoresearch-docstring-patch= / =autoresearch-docstring-apply= に渡す。
Phase 2b で proposal batch の harvest 接続を追加した。worktree 作成、
bench 実行、採用 commit 化は後続 Phase に残す。

** Phase 2 — eval-holdout + retroactive-revert gate (shipped)

- =autoresearch-docstring-apply= に =holdout_command= を追加
- 通常の =eval_command= は inner gate として扱う
- inner gate が keep した変更だけ holdout を実行する
- holdout が non-zero exit の場合は =keep_on_eval_failure= に関係なく
  元ファイル content を復元する
- 戻り値:
  - =:eval= inner eval 結果
  - =:holdout= holdout eval 結果
  - =:kept=
  - =:reverted=
  - =:revert-reason= (=eval-failed= / =holdout-failed= / nil)

** Phase 2a — docstring multi-candidate local loop (shipped)

- =autoresearch-docstring-run= MCP tool を追加
- =candidates_json= は JSON array:
  - string entry: そのまま replacement description
  - object entry: =description= / =new_description= と optional =label=
- 各候補は同じ original snapshot に適用してから =eval_command= を実行
- inner eval が通った候補だけ =holdout_command= を実行
- holdout まで通った候補は、仮の description に対する docstring issue score
  (=short-description= / =generic-description= / usage guidance 等) で採点
- 全試行後に source を original へ戻し、最良候補だけを final apply する
- 全候補が gate を通らなければ original を維持し、=:kept nil= を返す
- 戻り値:
  - =:attempts= candidate ごとの eval / holdout / reject reason / post-score
  - =:selected= 採用候補
  - =:kept= / =:reverted=

この Phase は worktree を作らない。既存 apply/eval/holdout primitive の上に
安全な多候補 local 探索を載せる段階で、worktree 横断・orchestrator batch
polling・採用 commit 化は次段に残す。

** Phase 2b — proposal harvest bridge (shipped)

- =autoresearch-docstring-harvest= MCP tool を追加
- =batch_id= の orchestrator batch を =anvil-orchestrator-collect= で回収
- optional =wait= が truthy なら batch terminal まで待ってから collect
- =status= が =done= で =summary= が non-empty の task だけを candidate 化
- fenced block や quoted string だけの summary は description text へ正規化
- 候補が無い場合は source file を編集しない
- 候補がある場合は JSON array にして =autoresearch-docstring-run= へ渡し、
  eval / holdout / issue-score で最良 proposal だけを残す
- 戻り値:
  - =:results= collect した slim task list
  - =:candidates= harvested description candidates
  - =:run= =autoresearch-docstring-run= の結果
  - =:kept= / =:reverted=

これで Q1-A の docstring autoresearch は、提案生成を orchestrator に出し、
batch 結果を回収して local eval/holdout gate に流す基本 loop まで接続済み。
Phase 2c で proposal batch の N 本生成 helper も追加済み。Phase 2d で
worktree 分離の明示制御を proposal task metadata に接続済み。Phase 2e で

Q1-B への拡張。

** Phase 2c — proposal batch fan-out helper (shipped)

- =autoresearch-docstring-propose-batch= MCP tool を追加

- =count= 本の orchestrator task を同一 batch に submit
- task name は =autoresearch-docstring-<tool>-NN-of-MM= で一意化
- prompt には =candidate N of M= を入れて、候補間の variation を促す

  - =:budget-usd=
  - =:timeout-sec=
  - =:heartbeat-timeout-sec=
  - =:cwd=
  - =:sandbox=
- 戻り値:
  - =:batch-id=
  - =:count=
  - =:tasks= 実際に submit した task list
  - =:plan=

運用 sequence は:

1. =autoresearch-docstring-propose-batch= で N 候補を async submit
2. =orchestrator-status= で batch を観察
3. =autoresearch-docstring-harvest= で done summaries を回収
4. =autoresearch-docstring-run= が eval / holdout / issue-score で採用判断

** Phase 2d — explicit worktree controls (shipped)

- =autoresearch-docstring-propose= に =no_worktree= / =worktree_name= を追加
- =autoresearch-docstring-propose-batch= に =no_worktree= / =worktree_name= を追加
- =no_worktree= truthy の時は task plist に =:no-worktree t= を載せる

これで Doc43 は proposal fan-out 時点で worktree isolation を明示できる。

* Primitive vs Convention 分離

scope GENERIC として ship する場合:

- *Primitive (一般化可能)*:
  - loop state machine
  - propose → apply → eval → decide サイクル
  - holdout gate
  - budget hard stop
  - snapshot/restore via anvil-session

- *Convention (運用者固有)*:
  - 最適化対象の選定 (MCP docstring / CLAUDE.md / ...)
  - 評価関数の定義 (make test / anvil-bench / custom)
  - skill promote 時の anvil-memory schema

→ primitive 層は anvil-autoresearch.el に閉じ、convention 層は
=anvil-autoresearch-target-spec= 構造体で declarative に与える形に
すれば外部公開可能。
