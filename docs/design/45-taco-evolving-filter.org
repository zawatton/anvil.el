論文の核心:

- ターミナルエージェント出力履歴は O(n²) で膨張、後半 1 ステップで数十万
  トークン消費
- Lost-in-the-Middle で中盤エラーログを見落とす
- 既存の static filter (正規表現/手書きルール/学習型) は固定圧縮器で、
  新ツール追加に追随できず破綻
- TACO は **進化する圧縮機**: タスク実行中に観測分布へ動的適応
- アブレーション: 進化を除去すると -1.7pp、グローバルプール除去は -0.2pp
  → 「事前蓄積より動的進化が支配的」

* Motivation

Doc 27 (anvil-shell-filter) は静的 filter table (60 件) で実装済:

- =git status= / =rg= / =pytest= 等は強力に効く (rtk 由来で実証)
- *未知コマンド* (例: 新しい CLI、業務固有 script) は素通し
- 新コマンド追加には手で filter を書く必要あり (Phase 4 業務固有 filter)

TACO のアプローチを取り込めば:

1. 未知コマンドの出力を **その場でルール生成** して圧縮
2. 過剰圧縮 (エージェントが「フルアウトプット見せて」と要求 / 同コマンド
   再実行) を検知して自動緩和
3. Critical 行 (error / stack trace) は絶対保持
4. Tool Attention (Doc 44) が context 入口の選別、TACO は出口の圧縮、
   組み合わせで累乗効果

論文実証データ:

- 7-Zip 展開ルール自動発見: 保持は error 行 + "Everything is OK" のみ、
  124 回適用で苦情 0・信頼度 1.0
- objdump 逆アセンブル: 圧縮率 0.146 (元の 15%)、苦情なし
- ターミナル外 (SWE-bench Lite +0.82pp / Devin-Eval +1.64pp / Rust-bench
  +1.05pp) でも効果

* Open Questions (要議論)

** Q1. Layer 1 だけ取るか、Layer 2/3 まで取るか

論文の 3 層構成:

- *Layer 1* — 出力圧縮 (Critical 保持 + ノイズ削除)
- *Layer 2* — タスク内ルール進化 (未知出力 → LLM ルール生成、過剰圧縮
  検知で保守的修正)
- *Layer 3* — グローバルプール (タスク横断、信頼度閾値で昇格、RDNTK 収束
  判定)

アブレーション結果: Layer 3 除去 -0.2pp (ほぼ無影響)、Layer 2 除去 -1.7pp。

→ Layer 1 + 2 が essential、Layer 3 は marginal。

選択肢:

- (a) Layer 1 のみ取り込み (静的圧縮を anvil-shell-filter で既に持つので
      ROI 低い)
- (b) Layer 1 + 2 取り込み (動的進化が本体、~3-5日 → ~1-2週に拡大)
- (c) 全 3 層

→ **推奨**: (b) Layer 1+2。Layer 1 単独だと既存 anvil-shell-filter と
   差別化が薄い。Layer 2 (進化) が論文の core value。

(変更: 元マトリクス「Layer 1 のみ ~3-5日」前提を Layer 1+2 ~1-2週に
revise。)

** Q2. ルール生成 LLM をどこに置くか

論文ではタスク内 LLM がルール生成。anvil 文脈:

- (a) anvil-orchestrator codex 経路 (Pro plan 既存、規約 OK)
- (b) tiny local model (HF transformers 等)
- (c) main Claude session 自身に「このコマンド出力のルールを書け」

→ **推奨**: (a) codex。orchestrator で既に subprocess 経路ある。
   レイテンシ (10-30s) は許容範囲 (1 コマンドの初回のみ)。

** Q3. 過剰圧縮検知 (= 苦情シグナル)

論文の苦情検知パターン:

1. エージェントが "show me the full output" を要求
2. 同じコマンドを引数同じで再実行する
3. session-events に "review tool call" が連続出現

→ anvil 文脈:

- (1) → Claude transcript の正規表現 detector
- (2) → anvil-shell-tee の hit 履歴で重複検出
- (3) → anvil-session-events で連続パターン検出

3 シグナルすべて anvil 既存 primitive で取れる。

** Q4. ルール保持期間 (TTL)

論文: グローバルプールは信頼度 + 使用実績で promote/demote。

anvil 文脈での TTL 案:

- session-local rule: session 終了で破棄 (= Layer 2 のみ)
- worktree-local rule: worktree 削除で破棄
- global rule: anvil-state 永続化、信頼度 < 0.5 で 30日後 GC

→ 初手は session-local + worktree-local のみ。global は Phase 3 (= TACO
   Layer 3) で。

** Q5. 既存 anvil-shell-filter との競合解決

同じコマンド (例: =git status=) に対して:

- anvil-shell-filter (Doc 27) は static rule で 600 トークンに圧縮
- anvil-taco は session-local rule を生成しようとする

どちらが勝つか?

→ **推奨**: anvil-shell-filter (static) が優先、anvil-taco は未対応
   コマンドで fallback。Doc 27 の dispatcher が "no filter" を返した
   時のみ TACO 経路に入る設計。

** Q6. Critical 行の判定基準

論文では "error" / "stack trace" を絶対保持。

anvil 文脈での Critical 判定:

- 標準 stderr 出力すべて
- =Error:= / =Traceback= / =FAILED= / =fatal:= で始まる行
- exit code が non-zero 時の最終 20 行
- ユーザー定義 regex (defcustom =anvil-shell-filter-taco-critical-patterns=)

* Phase 分割案

| Phase | scope                                                     | 工数 |
|-------+-----------------------------------------------------------+------|
| 1     | Layer 1 のみ (critical-keeper + fallback すべき素通し)    | 小   |
|       | Doc 27 dispatcher 未対応コマンドで critical-keeper だけ   |      |
|       | 動かす。圧縮は弱いが安全。                                |      |
| 2     | Layer 2 rule-evolver (LLM 呼出し + session-local store)   | 中   |
| 3     | complaint-detector + rule-relaxer (進化の本体)            | 小   |
| 4     | worktree-local rule の永続化                              | 小   |
| 5     | Layer 3 global rule + RDNTK 収束判定 (Phase 2 後 marginal | 中   |
|       | gain 確認してから着手)                                    |      |

* 評価メトリクス

- 圧縮率 = compressed_tokens / raw_tokens (target ≤ 0.3)
- 苦情率 = complaints / total_compressions (target < 5%)
- per-session 累積トークン (anvil-state-tracked)
- ルール生成コスト (codex API tokens / rule)

* 安全策

- Critical 保持は **rule-evolver より優先**。LLM が "remove error line"
  と言っても無視。
- 1 コマンド初回は LLM 呼出し overhead あり (10-30s)、cache miss 警告を
  emit。
- ルール暴走防止: 同一コマンドで 3 回連続「苦情→緩和」が起きたら static
  fallback (= 素通し) に切替。
- LLM 生成ルールは sandbox eval (eval されない、正規表現 / 行範囲指定の
  declarative DSL のみ)。

* 取り込み見送り判断材料

- Phase 1 で critical-keeper の絶対保持が機能する (false negative=0)
- Phase 2 で初回コスト ≤ 30s、再利用率 ≥ 70%
- Phase 3 で苦情率 ≤ 5% に収束
