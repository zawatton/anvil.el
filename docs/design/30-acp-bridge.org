* Motivation

** r/emacs landscape signal (2026-04-21)

theeseuus の MCP-for-Lisp 周辺 landscape リストに ~agent-shell
(xenodium)~ が含まれ、ACP (Agent Client Protocol, Zed + Google 2025 策定)
への露出が拡大している事実が確認された:

| Signal                                    | 値                    |
|-------------------------------------------+-----------------------|
| agent-shell stars                         | 1,060                 |
| ACP editor support (as of 2026-04)        | Zed / JetBrains / Neovim / Emacs |
| ACP spec site                             | agentclientprotocol.com |
| MCP forwarding in ACP                     | 標準サポート          |

agent-shell は Emacs frontend / anvil は Emacs infrastructure という構造。
frontend が MCP server を agent 経由で利用するフローに anvil が乗れていないと、
"anvil の tool registry は MCP client (Claude Code) からしか使えない" と
いう distribution のボトルネックが残る。

** MCP と ACP の役割分担

#+begin_src text
[MCP]  Model  ─────→  Tools/Data
                     (anvil-file / anvil-git / anvil-http / ...)

[ACP]  Editor ─────→  Agent
       (Emacs/Zed)   (Claude Code / Gemini CLI / Goose / anvil?)
                      │
                      └──→ MCP Tools (forwarded)
#+end_src

ACP agent は通常 *LLM 呼び出しを内包した subprocess* (Claude Code CLI,
Gemini CLI 等) で、editor からの prompt を受けて自分の判断で MCP tools を
叩く。anvil を ACP agent 化すると:

- anvil-orchestrator の 5 provider を ACP editor 側から指名できる
- anvil-consensus (判定投票) を ACP 経由で呼べる
- anvil の MCP tool 群 (file/git/http/buffer/org/pty-broker) を agent
  self-use する

つまり anvil の "multi-provider + consensus" という差別化要素を ACP
editor 群に expose できる。

** なぜ今か

- ACP は 2025-08 に Zed + Google がアナウンス、2026-04 時点で editor 側
  採用が加速中。*先行期* に入っているので agent 側の選択肢は少ない。
- agent-shell が Emacs 側 ACP client の事実上の定番になりつつあるが、
  ACP agent として名指しされているのは Claude Code / Gemini CLI /
  Goose / Cline 等の *商用 CLI* が中心。OSS Elisp 製 agent は空白。
- anvil は既に worker pool + orchestrator + streaming を持っており、
  不足は *protocol wrapper* のみ。追加工数 ROI が高い。

* Goals

1. *=anvil-acp=* モジュールを新規作成。opt-in (=anvil-optional-modules=)。
2. *ACP spec 準拠の stdio JSON-RPC server* を起動するトップコマンド
   =anvil-acp-serve= を提供 (=M-x= / CLI 両対応)。
3. 既存 =anvil-register-tools= をそのまま re-expose — MCP と ACP で
   tool 定義の二重化を *しない*。
4. anvil-orchestrator の streaming infra (token delta / status update)
   を ACP の =session/update= notification に bridge。
5. *MCP forward* — ACP 仕様の "editor が提供する MCP servers を agent に
   渡す" 機構をサポート。受け取った MCP server 設定は anvil 内の既存
   MCP client で透過的に扱う。
6. Phase 1 minimum: chat + tool-call + streaming deltas。deep agentic
   loop は Phase 2。

* Non-goals

1. anvil を ACP editor 化しない (agent-shell がその役目)。
2. ACP spec の full surface を初期から網羅しない — session resume /
   permission flow 等は Phase 2+。
3. 既存 MCP server としての anvil を ACP 統合のために破壊しない。

* Design Notes

** Transport / framing

ACP は MCP と同じく JSON-RPC 2.0 over newline-delimited stdio。anvil
側は既存の =anvil-mcp-transport= が newline-JSON を扱えるので、stream
layer は大部分再利用可能。

** Session lifecycle

#+begin_src text
editor                             anvil-acp
  │                                    │
  ├── initialize ────────────────────→ │
  │←── server capabilities ─────────── ┤
  ├── session/new ───────────────────→ │ (新 session ID 発行)
  │←── session ready ───────────────── ┤
  ├── prompt "..." ──────────────────→ │ (orchestrator submit)
  │                                    │ ┌─ LLM streaming ─┐
  │←── session/update (delta) ──────── ┤ │                 │
  │←── session/update (delta) ──────── ┤ │                 │
  │←── session/update (tool_call) ──── ┤ │                 │
  │                                    │ │ tool execution  │
  │←── session/update (tool_result) ── ┤ │                 │
  │←── prompt/response (final) ─────── ┤ └─────────────────┘
#+end_src

session ID は =anvil-orchestrator= の既存 session infra で管理できる
(Doc 10 Phase 6C の DAG session を再利用)。

** Tool dispatch

editor 側から MCP server を提供された場合、anvil-acp は:

1. MCP server 設定 (stdio command 等) を受け取り、
2. =anvil-mcp-client= 側で spawn し、
3. LLM が tool_call を emit したら該当 MCP server へ forward、
4. 結果を =session/update= で editor に流す。

anvil 内蔵の tool 群 (file / git / http / buffer / org ...) は同じ
bridge 経由で LLM から見える。MCP と ACP の境界を透過化できるのが
実装上の肝。

** Provider 選択

ACP agent = 1 LLM という前提が現行 editor には残っているため、Phase
1 では =anvil-acp-default-provider= (defcustom) で 1 個に固定する。

Phase 3 で anvil-consensus を ACP tool として expose し、LLM が
自分で "consensus(task)" を呼べば複数 provider の投票結果が返る形に。

* Open Questions

** Resolved (2026-04-22 research)

1. *ACP spec stability* — 解決。formal stability guarantee はまだ無い。
   version は =initialize= で negotiate、RFD process で改定。agent-shell
   側は 13 agent を既に吸収しているが、それは各 CLI 固有 adapter 扱い。
   anvil-acp は =initialize= の ~protocolVersion~ を固定宣言し、不一致な
   editor には明示エラーを返す方針で十分。

2. *MCP forward envelope* — **spec で未採択**。「MCP-over-ACP: MCP Transport
   via ACP Channels」は RFD 段階で、stable spec に入っていない。Phase 2a
   は spec 待ち。当面は anvil 内蔵 tool のみを expose する方針に変更。

3. *agent-shell adapter 要否* — 不要と確認。agent-shell は
   =agent-shell-<agent>-acp-command= (string list) で spawn するだけ。
   anvil-acp を spawnable CLI (例: =emacs -Q -L ... -l anvil-acp
   --batch -f anvil-acp-serve=) として提供すれば、user 側の config
   追加だけで自動対応する。adapter module 一切不要。

** 新規 open (2026-04-22)

4. *Process 構成* — anvil-acp は (a) 新規 emacs subprocess を毎 session
   起動するか、(b) 既存 daemon に stdio 経由で bridge するか。(a) は
   簡単だが毎回 1-2s の起動 cost、(b) は安いが bridge 実装が重い。
   Phase 0 で両方の PoC 比較を挟む。

5. *Multi-provider 露出* — anvil の強みは "consensus across providers"
   だが ACP 1 session = 1 provider 前提。consensus は tool として
   expose (Phase 3) するか、session/mode で切替えるか。両案の UX 検証
   を Phase 0 後段で実施。
