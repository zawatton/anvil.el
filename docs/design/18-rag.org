*Lexical core shipped* (2026-06-20) as the optional ~semantic~
module (~anvil-semantic.el~), replacing an earlier Python
(~semantic.py~, Gemini/OpenAI embeddings) prototype.  It is a pure
Emacs-Lisp retrieval engine over a configurable corpus of org /
text / code files (~anvil-semantic-roots~), backed by the built-in
SQLite *FTS5* index with the ~trigram~ tokenizer for CJK substring
matching.  No Python, no embeddings, no cloud.  Tools:
~semantic-search~ (FTS5 BM25 + lexical term-overlap, RRF-fused),
~notes-lexical-search~ (term-overlap + caller synonym expansion for
retrieve-then-read), ~semantic-reindex~, ~semantic-status~.

Key implementation note: FTS5 trigram MATCH needs ≥3 characters, so
retrieval *unions* an FTS5 MATCH pass (≥3-char terms; indexed,
BM25-ranked) with a ~LIKE~-scan pass (2-char kanji compounds —
整定/漏電/接地 — which trigram MATCH cannot reach), then ranks the
merged candidate set in Lisp.  The root-prefix filter uses
~instr(file, ?) = 1~ rather than ~LIKE~, because ~LIKE~/~GLOB~ do
not match on FTS5 *UNINDEXED* columns (only ~=~ and ~instr~ do).

The *embedding* layer (Option B) is now *shipped as an opt-in
bolt-on* (2026-06-20): a ~chunk_vec~ table joins on the stored
~digest~, built by the ~semantic-embed-index~ tool.  Provider
~anvil-semantic-embed-provider~ defaults to *ollama* (local, zero
egress); ~gemini~ / ~openai~ are opt-in cloud (API key via env or
~<db-dir>/<provider>.key~).  Search fuses three signals — FTS5
BM25, lexical overlap and embedding cosine — via RRF (modes:
hybrid / semantic / lexical / vector).  With an empty ~chunk_vec~
the engine is identical to the pure-FTS5 core, so the lexical
substrate and the retrieve-then-read flow remain the default.
Vectors are stored normalized as ~prin1~ float vectors in TEXT, so
cosine is a plain Lisp dot product (no numpy / no f32 unpack).

* Motivation

Refactors and "where do we already do something like this?" questions
don't reduce to a keyword.  Grep misses paraphrases; defs-search is
symbol-exact.  An LLM has the concept-matching, but every query burns
tokens.

A local embedding index — small model, batched — gives sub-100ms
concept-similarity over the whole project at near-zero marginal cost.
Vectors live in SQLite (blob column) alongside the same indexing
infrastructure Doc 02 / Doc 11 already use.

* Goals

1. Embedding index of elisp defs, org sections, design docs.
2. Model: ~nomic-embed-text~ via ollama or similar, locally hosted.
3. ~rag-search QUERY :k 5~ returns top-K (source + snippet + score).
4. Incremental update on file save (piggyback on Doc 11 / Doc 02
   filenotify watchers).

* Non-goals

- Online embedding (OpenAI, Cohere).  Too expensive at indexing scale.
- Chat-style RAG generation.  We return snippets; caller decides.
- Cross-modal (image / PDF).

* Architecture

#+begin_src sql
  CREATE TABLE rag_docs (
    id       INTEGER PRIMARY KEY,
    source   TEXT,        -- file path + section
    kind     TEXT,        -- 'defun' / 'org-headline' / 'design-section'
    content  TEXT,
    embedding BLOB        -- 768-dim f32
  );
  -- FTS5 virtual table for fallback keyword ranking
#+end_src

Query:
1. Embed query via local model.
2. Top-K cosine over all rows (brute force; <10k rows is fine).
3. Return snippets + score.

Ollama is the default embedding provider; swappable via defcustom.

* Phases

** Phase 1 — embedding infra + search

Ollama-only backend, elisp defs + design docs as sources.  ERT 5.

** Phase 2 — org sections + incremental update

Add org-headline as source kind, hook into Doc 02 watcher.

** Phase 3 — hybrid scoring

Combine embedding similarity with FTS5 BM25.  A simple weighted sum
usually outperforms either alone.
