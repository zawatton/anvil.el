** Why a separate module / DB

- ~capture/ai-logs-<platform>-<hostname>-YYYY.org~ は Claude 専用の append-only ログ。
  user は触らず、Linux / Windows で物理的に分離した別ファイルに書く。
- 既存の anvil-memory は「.md 横断知識ベース (永続)」が責務。worklog は
  「時系列 / machine 別の作業実績ログ」で、prune ロジック・ranking 概念
  (=decay-score=, =validity-prior= 等) も全く別物。
- 同じ DB に同居させると schema が肥大化し migrations が癒着する。
  別 DB (=Notes/.anvil-worklog/anvil-worklog-index.db=) で独立させる。
- ファイル本体 (org) は今まで通り canonical。SQLite は scan で再生成可能な
  index にすぎない (anvil-memory と同パターン)。

** Non-goals

- org → SQLite の双方向同期はしない。書き込みは Phase 2 の helper でも
  「org に append する」のが先で、index は scan で追従する。
- user 編集を許す journals-*.org / todo.org は対象外 — 触れない。
- 全文表示・editor integration は対象外 (Phase 3 で bridge 経由検討)。

* Phase 1 — read-only scan + MCP search/list

** Source format (既存、変更しない)

#+begin_src org
,#+title: AI 作業ログ 2026 (Linux/debian)
,#+date: 2026-01-01

,** NOTE 作業ログ <2026-04-20 Mon>
,*** MEMO AI: 案件名 <2026-04-20 Mon>
,**** 実施内容
,- ...
,**** 成果物
,- ...
,**** 次のアクション
,- ...

,*** MEMO AI: 別の案件 <2026-04-20 Mon>
... 同上
#+end_src

抽出単位は ~*** MEMO AI: <title> <date>~ section (level-3) 1 つ。
date は heading の inactive-timestamp、または親 ~** NOTE 作業ログ <date>~
からの fallback (level-3 に date が無い場合に備える)。

** Filename convention

~capture/ai-logs-<platform>-<hostname>-<year>.org~

例:
- ~capture/ai-logs-linux-debian-2026.org~ → machine = =linux-debian=, year = 2026
- ~capture/ai-logs-windows-ThinkPad-E14-Gen5-2026.org~ → machine = =windows-ThinkPad-E14-Gen5=, year = 2026

machine は filename から抽出する正規表現で得る:
~^ai-logs-\\(.+\\)-\\([0-9]\\{4\\}\\)\\.org$~

** SQLite schema

#+begin_src sql
CREATE TABLE IF NOT EXISTS worklog_entry (
  file        TEXT    NOT NULL,
  start_line  INTEGER NOT NULL,
  end_line    INTEGER NOT NULL,
  machine     TEXT    NOT NULL,
  year        INTEGER NOT NULL,
  date        TEXT    NOT NULL,            -- ISO "YYYY-MM-DD"
  title       TEXT    NOT NULL,
  body        TEXT    NOT NULL,
  digest      TEXT    NOT NULL,            -- sha1(body), used for upsert
  scanned_at  INTEGER NOT NULL,
  PRIMARY KEY (file, start_line)
);

CREATE INDEX IF NOT EXISTS worklog_entry_date_idx
  ON worklog_entry(date);

CREATE INDEX IF NOT EXISTS worklog_entry_machine_idx
  ON worklog_entry(machine);

CREATE VIRTUAL TABLE IF NOT EXISTS worklog_fts
  USING fts5(file UNINDEXED, start_line UNINDEXED,
             machine UNINDEXED, date UNINDEXED,
             title, body, tokenize='trigram');
#+end_src

CJK substring が引きたいので tokenize='trigram' を default。
SQLite が trigram 非対応だった場合は anvil-memory と同じく unicode61 に fallback。

** DB path resolver (anvil-memory と同型)

順序:
1. ~ANVIL_WORKLOG_DB~ env (set されていれば最優先)
2. =anvil-worklog-db-path= が default 以外に customize されていればそれ
3. =anvil-worklog-shared-db-roots= の各 root 配下に
   =.anvil-worklog/anvil-worklog-index.db= が **既に存在** すればそれ
4. default: =<user-emacs-directory>/anvil-worklog-index.db=

`(anvil-worklog-effective-db-path)` で実効パスを返す public API を提供。

** Scan algorithm

- 各 root (default = =capture/= ディレクトリ) を =directory-files= で歩く
- ファイル名の正規表現 ~ai-logs-.+-[0-9]{4}\\.org$~ にマッチするものだけ対象
- ファイル毎に temp-buffer で UTF-8 で開き、=org-mode= に入れず regex 走査
  - level-3 ~^\\*\\*\\* MEMO AI:~ の position を順に取得
  - 各 entry の終端は次の同等 level (~^\\*\\*\\* ~) または上位 (~^\\* ~ / ~^\\*\\* ~)
  - title = 行末の inactive-timestamp 直前まで
  - date = 行末の =<YYYY-MM-DD ...>= から抽出。失敗時は親 ~** NOTE 作業ログ
    <DATE>~ から fallback。両方失敗時は entry を skip (warn-only)
- digest = sha1(body) で upsert: 既存行の digest と一致なら skip、
  異なれば DELETE + re-INSERT (worklog_fts も同期)
- 戻り値は plist =(:files N :entries M :inserted I :updated U :unchanged S)=

`anvil-memory-scan` と違い、source ファイル数 = ai-logs-*.org の数だけ
(普通 1–3 個) なので scan は msec オーダー。

** Public API

| Name                                   | Returns                                                     |
|----------------------------------------+-------------------------------------------------------------|
| =anvil-worklog-effective-db-path=      | path string                                                 |
| =anvil-worklog-scan &optional roots=   | plist counts                                                |
| =anvil-worklog-prune &optional roots=  | int (rows deleted for missing files)                        |
| =anvil-worklog-search QUERY &key …=    | list of entry plists (FTS5 matched)                         |
| =anvil-worklog-list &key …=            | list of entry plists (date DESC)                            |
| =anvil-worklog-get FILE START-LINE=    | entry plist or nil                                          |

Search / list は共通のフィルタ:
- =:limit= INT (default 20)
- =:machine= STRING (filename machine token と equal 比較)
- =:since= ISO date — 含む
- =:until= ISO date — 含む
- =:year= INT

Entry plist の shape:
=(:file path :start-line N :end-line M :machine "linux-debian"
  :year 2026 :date "2026-04-29" :title "anvil-worklog Phase 1"
  :body "..." :digest "...")=

** Schema impact

新規 DB のため migration 不要。既存 anvil-memory DB には触らない。

** Risks

- *R1: 巨大 ai-logs ファイルでの scan コスト*: 14000 行を超える linux-debian
  ログでも regex 走査は msec オーダー (実測予定)。万一遅ければ
  =mtime + size= を file ごとに記録して skip する optimization を Phase 1.5 で追加。
- *R2: org timestamp 形式バリエーション*: =<2026-04-29 Wed>= / =<2026-04-29>=
  / =<2026-04-29 Wed 12:00>= の 3 形式を許容。曜日や時刻は捨てて ISO date のみ抽出。
- *R3: machine 衝突*: 異なる物理マシンで同一 hostname を使うと混ざるが、
  CLAUDE.md の命名規約 (=hostname -s=) で実用的に衝突しない。明示衝突検出は不要。
- *R4: digest 衝突 (sha1 collision)*: 実用上ゼロリスク扱い。

* Phase 2 — write helper (deferred)

`(anvil-worklog-append-memo TITLE BODY &key date file)` で
~** NOTE 作業ログ <DATE>~ section の自動探索 + 無ければ作成 + ~*** MEMO AI:~ 追記、
